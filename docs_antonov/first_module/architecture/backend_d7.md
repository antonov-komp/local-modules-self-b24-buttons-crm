## Архитектура backend (D7) модуля «BP Button Field»

Backend модуля реализуется на D7 и обеспечивает:

- регистрацию пользовательского типа поля `bp_button_field`;
- хранение и выдачу настроек кнопок;
- обработку событий ядра, связанных с пользовательскими полями;
- безопасный AJAX‑API для фронтенда (JS‑расширения).

Архитектура опирается на требования из `requirements.md`.

---

### 1. Основные компоненты backend

1. **ORM‑сущность настроек кнопок**
   - Класс: `My\BpButton\Internals\SettingsTable` (наследник `Bitrix\Main\Entity\DataManager`).
   - Таблица: `my_bpbutton_settings`.
   - Назначение:
     - хранение конфигурации логики для каждого пользовательского поля типа `bp_button_field`;
     - связь с системной таблицей `b_user_field` через `FIELD_ID`.

2. **Контроллер для AJAX‑запросов**
   - Класс: `My\BpButton\Controller\ButtonController`
     (наследник `Bitrix\Main\Engine\Controller`).
   - Назначение:
     - приём запросов от JS‑расширения при клике по кнопке;
     - проверка сессии и прав доступа пользователя;
     - возврат конфигурации `SidePanel` (URL, заголовок, ширина) с учётом контекста CRM.

3. **Сервисный слой**
   - Класс: `My\BpButton\Service\ButtonService` (рабочее название).
   - Назначение:
     - инкапсуляция бизнес‑логики:
       - поиск и валидация настроек по `FIELD_ID`;
       - формирование DTO/массива с настройками для контроллера;
       - потенциальная интеграция с логированием и условиями отображения.

4. **Обработчик событий модуля**
   - Класс: `My\BpButton\EventHandler` или набор отдельных классов в `Service/` / `Handler/`.
   - Назначение:
     - регистрация типа `bp_button_field` в событии `OnUserTypeBuildList`;
     - реакция на создание/удаление пользовательских полей:
       - `OnAfterUserFieldAdd`;
       - `OnUserFieldDelete`.

---

### 2. Пользовательский тип поля `bp_button_field`

Регистрация кастомного типа поля выполняется через обработчик события:

- **Событие:** `main:OnUserTypeBuildList`.
- **Реализация:** метод в `EventHandler`, возвращающий массив описания пользовательского типа.

Ключевые элементы описания user type:

- `USER_TYPE_ID` → `bp_button_field`;
- `CLASS_NAME` → PHP‑класс, реализующий логику отображения/редактирования (если требуется);
- `DESCRIPTION` → локализованное название типа поля;
- `GetDBColumnType` → тип колонки в БД (как правило, строка для хранения ID/служебного значения);
- `GetEditFormHTML` / `GetAdminListViewHTML` и др. — при необходимости;
- `GetPublicViewHTML` — возвращает HTML кнопки для отображения в карточке CRM.

**Требование:** в публичном представлении (`view`) поле должно отрисовываться
как кнопка с нужным CSS‑классом (из настроек поля), с атрибутами/данными,
позволяющими JS‑расширению идентифицировать:

- `ENTITY_ID`;
- `ELEMENT_ID`;
- `FIELD_ID`;
- при необходимости — дополнительные параметры.

---

### 3. Обработка событий ядра

#### 3.1. `OnUserTypeBuildList`

- Цель: зарегистрировать тип пользовательского поля `bp_button_field` в системе.
- Где: метод класса `EventHandler`, подключаемый через `init.php` модуля или `install/index.php`.
- Результат: в интерфейсе добавления пользовательского поля в CRM появляется новый тип поля.

#### 3.2. `OnAfterUserFieldAdd`

- Цель: при создании поля типа `bp_button_field` автоматически создать запись в
  `my_bpbutton_settings`.
- Логика:
  - проверить, что добавляемое поле относится к CRM‑сущности и имеет тип `bp_button_field`;
  - создать запись в `SettingsTable` с минимумом данных:
    - `FIELD_ID`;
    - базовые дефолтные настройки (URL, ширина, заголовок могут быть пустыми или шаблонными);
  - связать запись с сущностью/таблицей CRM при необходимости (через доп. поля).

#### 3.3. `OnUserFieldDelete`

- Цель: при удалении пользовательского поля типа `bp_button_field`
  каскадно удалить его настройки.
- Логика:
  - по `FIELD_ID` удалить записи из `my_bpbutton_settings`;
  - в будущем — дополнительно чистить связанные логи из `my_bpbutton_logs`.

#### 3.4. Схема работы обработчиков событий

Упрощённая последовательность:

1. **Инициализация модуля:**
   - модуль подключается (`init.php`/`install/index.php`);
   - регистрируются обработчики событий (`OnUserTypeBuildList`, `OnAfterUserFieldAdd`, `OnUserFieldDelete`).
2. **Администратор создаёт поле `bp_button_field` в CRM:**
   - ядро сохраняет настройки поля в `b_user_field`;
   - срабатывает `OnAfterUserFieldAdd` модуля:
     - проверяет, что `USER_TYPE_ID = bp_button_field`;
     - создаёт запись в `SettingsTable` с `FIELD_ID` и дефолтными значениями настроек.
3. **Администратор настраивает логику в админ‑реестре:**
   - обновляются поля `HANDLER_URL`, `TITLE`, `WIDTH` в `my_bpbutton_settings`.
4. **Удаление поля:**
   - при удалении пользовательского поля ядро вызывает `OnUserFieldDelete`;
   - обработчик модуля удаляет соответствующие записи из `SettingsTable` (и, при необходимости, логов).

### 4. Контроллер `ButtonController` (Bitrix\Main\Engine\Controller)

Контроллер предоставляет публичные методы для фронтенда (JS):

- Базовый endpoint: `my.bpbutton:button` (рабочее имя, уточняется при реализации).
- Пример метода:
  - `public function getConfigAction($entityId, $elementId, $fieldId): array`
    - входные параметры:
      - `entityId` — тип сущности (лид, сделка и т.п.);
      - `elementId` — ID элемента;
      - `fieldId` — ID пользовательского поля;
    - шаги:
      1. `check_bitrix_sessid()` — проверка сессии;
      2. проверка прав пользователя через CRM‑авторизацию;
      3. запрос настроек из `ButtonService` / `SettingsTable`;
      4. формирование ответа c:
         - `url` — URL для iframe;
         - `title` — заголовок слайдера;
         - `width` — параметры ширины (px/%);
         - `context` — сущность/элемент/пользователь;
      5. возврат массива, который Engine преобразует в JSON.
- Обработка ошибок:
  - при отсутствии прав/настроек или ошибках валидации контроллер возвращает
    структурированный ответ об ошибке (код + сообщение), без раскрытия внутренних деталей.

#### 4.1. Общая схема вызова AJAX‑контроллера

1. Пользователь открывает карточку CRM с полем `bp_button_field`.
2. User type `bp_button_field` отрисовывает кнопку с `data-*` атрибутами.
3. При клике JS‑расширение:
   - ставит кнопку в состояние `state-wait`;
   - формирует запрос к `ButtonController.getConfigAction` с параметрами:
     `entityId`, `elementId`, `fieldId`, `sessid`.
4. Контроллер:
   - проверяет сессию (`check_bitrix_sessid`);
   - проверяет права доступа пользователя;
   - через `ButtonService` получает настройки из `SettingsTable`;
   - возвращает JSON с конфигурацией SidePanel или ошибкой.
5. JS‑код:
   - при успехе открывает `BX.SidePanel` с указанным URL/шириной/заголовком;
   - при ошибке показывает уведомление и возвращает кнопку из `state-wait`.

---

### 5. Сервисный слой `ButtonService`

`ButtonService` отвечает за бизнес‑логику вокруг кнопок:

- Получение настроек по `FIELD_ID`:
  - валидация существования записи в `SettingsTable`;
  - объединение данных из таблицы настроек и контекста CRM‑элемента.
- Подготовка конфигурации для `SidePanel`:
  - применение значений по умолчанию при отсутствии настроек;
  - учёт будущих условий отображения (стадия сделки, значения других полей).
- Потенциальная интеграция с логированием:
  - фиксация факта запроса конфигурации;
  - регистрация результата в таблице логов (в следующих версиях).

Сервисный слой позволяет изолировать бизнес‑логику от кода контроллера и облегчить тестирование.

---

### 6. Связь backend с frontend

1. **Отрисовка кнопки:**
   - backend (user type) генерирует HTML кнопки с нужными `data-*` атрибутами:
     - `data-entity-id`, `data-element-id`, `data-field-id`, `data-user-id`.
2. **Клик по кнопке:**
   - JS‑код (описан в `frontend_ui.md`) читает атрибуты и вызывает метод контроллера.
3. **Ответ контроллера:**
   - JSON с конфигурацией `SidePanel`.
4. **Открытие SidePanel:**
   - frontend открывает `BX.SidePanel.Instance` c URL/параметрами, полученными от backend.

Такой контракт обеспечивает слабую связанность и возможность изменения логики показа
или формирования URL на стороне backend без правки разметки карточки.
