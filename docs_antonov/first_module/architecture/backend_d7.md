## Архитектура backend (D7) модуля «BP Button Field»

Backend модуля реализуется на D7 и обеспечивает:

- регистрацию пользовательского типа поля `bp_button_field`;
- хранение и выдачу настроек кнопок;
- обработку событий ядра, связанных с пользовательскими полями;
- безопасный AJAX‑API для фронтенда (JS‑расширения);
- (в расширенной версии) логирование нажатий и результатов.

Архитектура опирается на требования из `requirements.md` и концепцию из `local_button.md`.

---

### 1. Слои backend и принципы разделения ответственности

Backend модуля условно делится на несколько слоёв:

1. **Слой данных (ORM / D7)**  
   Отвечает за работу с таблицами `my_bpbutton_settings`, `my_bpbutton_logs` и связь с системными таблицами (`b_user_field` и др.).  
   Здесь **нет бизнес‑логики**, только CRUD‑операции и валидация на уровне схемы.

2. **Сервисный слой (бизнес‑логика)**  
   Инкапсулирует всю бизнес‑логику вокруг кнопок:
   - интерпретация настроек;
   - подготовка конфигурации SidePanel;
   - принятие решений на основе контекста (будущие условия отображения);
   - интеграция с логированием.  
   Контроллеры и обработчики событий **не должны содержать сложную логику**, а только вызывать сервисы.

3. **Слой интеграции с ядром (события / user type)**  
   Включает:
   - регистрацию пользовательского типа `bp_button_field`;
   - реакцию на `OnAfterUserFieldAdd` / `OnUserFieldDelete`.  
   Здесь используется сервисный слой и ORM, но **не принимаются бизнес‑решения** на уровне UI.

4. **Слой контроллеров (HTTP/AJAX‑API)**  
   Реализуется через `Bitrix\Main\Engine\Controller` и отвечает за:
   - приём и валидацию параметров;
   - проверку сессии и прав;
   - вызов сервисного слоя;
   - возврат структурированных ответов (JSON).  
   Здесь **нет прямой работы с БД** и минимальная логика.

### 2. Слой данных: ORM‑сущности

#### 2.1. `SettingsTable` — настройки кнопок

- Класс: `My\BpButton\Internals\SettingsTable` (наследник `Bitrix\Main\Entity\DataManager`).
- Таблица: `my_bpbutton_settings`.
- Назначение:
  - хранение конфигурации логики для каждого пользовательского поля типа `bp_button_field`;
  - связь с `b_user_field` через `FIELD_ID`.
- Детали структуры и DDL приведены в `data_model.md`.

#### 2.2. `LogsTable` (планируемая сущность)

- Класс: `My\BpButton\Internals\LogsTable` (рабочее название).
- Таблица: `my_bpbutton_logs`.
- Назначение:
  - хранение истории нажатий кнопок, результатов и диагностических сообщений.
- Реализация и DDL также описываются в `data_model.md` и привязаны к `TASK-004-logging.md`.

---

### 3. Пользовательский тип поля `bp_button_field`

User type является точкой входа backend‑логики в интерфейс CRM.

#### 3.1. Регистрация типа

- **Событие:** `main:OnUserTypeBuildList`.
- **Реализация:** метод в `EventHandler`, возвращающий массив описания пользовательского типа.

Основные ключи массива:

- `USER_TYPE_ID` → `bp_button_field`;
- `CLASS_NAME` → PHP‑класс, реализующий user type (например, `My\BpButton\UserField\BpButtonUserType`);
- `DESCRIPTION` → локализованное название типа поля;
- `GetDBColumnType` → тип колонки в БД (как правило, строка для хранения служебного значения);
- визуальные методы (`GetEditFormHTML`, `GetAdminListViewHTML`, `GetPublicViewHTML` и др.).

#### 3.2. Публичное представление (view)

В публичном представлении (`GetPublicViewHTML`) поле должно отрисовываться как кнопка:

- использующий стиль из настроек (классы Bitrix UI Kit);
- с набором `data-*` атрибутов, которые читает JS‑расширение:
  - `data-entity-id` (`ENTITY_ID`);
  - `data-element-id` (`ELEMENT_ID`);
  - `data-field-id` (`FIELD_ID`);
  - `data-user-id` (`USER_ID`).

**Принцип:** user type не реализует бизнес‑логику, а только:

- подготавливает корректное представление;
- обеспечивает «контракт» с фронтендом (атрибуты, классы, идентификаторы).

### 4. Обработка событий ядра

#### 4.1. `OnUserTypeBuildList`

- Цель: зарегистрировать тип пользовательского поля `bp_button_field` в системе.
- Где: метод класса `EventHandler`, подключаемый через `init.php` модуля или `install/index.php`.
- Результат: в интерфейсе добавления пользовательского поля в CRM появляется новый тип поля.

#### 4.2. `OnAfterUserFieldAdd`

- Цель: при создании поля типа `bp_button_field` автоматически создать запись в `my_bpbutton_settings`.
- Логика (на уровне обработчика):
  - проверить, что добавляемое поле относится к CRM‑сущности и имеет тип `bp_button_field`;
  - вызвать метод сервисного слоя (например, `ButtonService::onFieldCreated`) и передать параметры поля;
  - сервис создаёт запись в `SettingsTable` с:
    - `FIELD_ID`;
    - дефолтными значениями настроек (URL/ширина/заголовок могут быть пустыми или шаблонными).

#### 4.3. `OnUserFieldDelete`

- Цель: при удалении пользовательского поля типа `bp_button_field` каскадно удалить его настройки.
- Логика:
  - по `FIELD_ID` вызвать сервис, который:
    - удалит записи из `my_bpbutton_settings`;
    - при необходимости — очистит связанные логи из `my_bpbutton_logs` (в будущих версиях).

#### 4.4. Схема работы обработчиков событий

1. **Инициализация модуля:**
   - модуль подключается (`init.php` / `install/index.php`);
   - регистрируются обработчики событий (`OnUserTypeBuildList`, `OnAfterUserFieldAdd`, `OnUserFieldDelete`).
2. **Администратор создаёт поле `bp_button_field` в CRM:**
   - ядро сохраняет настройки поля в `b_user_field`;
   - срабатывает `OnAfterUserFieldAdd` модуля;
   - сервисный слой создаёт дефолтную запись в `SettingsTable`.
3. **Администратор настраивает логику в админ‑реестре:**
   - через UI обновляются поля `HANDLER_URL`, `TITLE`, `WIDTH` в `my_bpbutton_settings`.
4. **Удаление поля:**
   - при удалении пользовательского поля ядро вызывает `OnUserFieldDelete`;
   - обработчик модуля передаёт `FIELD_ID` в сервис, который очищает `SettingsTable` (и логи).

### 5. Контроллер `ButtonController` (Bitrix\Main\Engine\Controller)

Контроллер предоставляет публичные методы для фронтенда (JS).

- Базовый endpoint: рабочее имя, например `my.bpbutton:button` (уточняется при реализации).

Типичный метод:

- `public function getConfigAction($entityId, $elementId, $fieldId): array`
  - входные параметры:
    - `entityId` — тип сущности (лид, сделка и т.п.);
    - `elementId` — ID элемента;
    - `fieldId` — ID пользовательского поля;
  - шаги:
    1. `check_bitrix_sessid()` — проверка сессии;
    2. проверка прав пользователя через CRM‑авторизацию;
    3. вызов сервисного слоя (`ButtonService`) для получения настроек;
    4. формирование ответа с:
       - `url` — URL для iframe;
       - `title` — заголовок слайдера;
       - `width` — параметры ширины (px/%);
       - `context` — сущность/элемент/пользователь;
    5. возврат массива, который Engine преобразует в JSON.

**Обработка ошибок:**

- при отсутствии прав/настроек или ошибках валидации контроллер возвращает
  структурированный ответ об ошибке (код + сообщение), без раскрытия внутренних деталей;
- при системных ошибках — логирование в логи модуля с маскировкой чувствительных данных.

Список action‑методов и форматы ответов детализируются в `api/ajax_controller.md`.

---

### 6. Сервисный слой `ButtonService`

`ButtonService` отвечает за бизнес‑логику вокруг кнопок:

- **Получение настроек по `FIELD_ID`:**
  - валидация существования записи в `SettingsTable`;
  - объединение данных из таблицы настроек и контекста CRM‑элемента (при необходимости).
- **Подготовка конфигурации для SidePanel:**
  - применение значений по умолчанию при отсутствии отдельных настроек;
  - учёт будущих условий отображения (стадия сделки, значения других полей).
- **Интеграция с логированием (расширение):**
  - фиксация факта вызова (кликов, ошибок);
  - запись результата в `LogsTable` в соответствии с `TASK-004-logging.md`.

**Принцип:** контроллеры и обработчики событий не содержат бизнес‑правил, а только оркестрируют вызовы `ButtonService`.

---

### 7. Связь backend с frontend

1. **Отрисовка кнопки:**
   - backend (user type) генерирует HTML кнопки с нужными `data-*` атрибутами:
     - `data-entity-id`, `data-element-id`, `data-field-id`, `data-user-id`.
2. **Клик по кнопке:**
   - JS‑код (описан в `frontend_ui.md` и `ui_ux/crm_card_button.md`) читает атрибуты и вызывает метод контроллера.
3. **Ответ контроллера:**
   - JSON с конфигурацией `SidePanel` (или ошибкой).
4. **Открытие SidePanel:**
   - frontend открывает `BX.SidePanel.Instance` c URL/параметрами, полученными от backend.

Такой контракт обеспечивает слабую связанность и возможность изменения логики показа
или формирования URL на стороне backend без правки разметки карточки.
