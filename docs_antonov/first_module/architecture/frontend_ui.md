## Frontend UI: контракт JS для модуля «BP Button Field»

Документ описывает технический контракт фронтенда с backend‑слоем:
- какие HTML‑элементы и `data-*` атрибуты генерирует user type;
- какие JS‑модули Bitrix используются;
- жизненный цикл кнопки: инициализация → клик → запрос → открытие SidePanel → обработка ошибок.

Ссылается на:
- `../ui_ux/crm_card_button.md` (UX‑аспекты);
- `../api/ajax_controller.md` (формат API);
- `backend_d7.md` (роль user type и контроллера).

---

### 1. HTML‑контракт: разметка кнопки

#### 1.1. Базовый вид HTML‑элемента

User type `bp_button_field` в публичном представлении (`GetPublicViewHTML` и аналоги) должен генерировать разметку следующего вида (концептуально):

```html
<button
  class="ui-btn ui-btn-primary js-bp-button-field"
  data-entity-id="DEAL"
  data-element-id="123"
  data-field-id="456"
  data-user-id="789"
>
  Дополнительные действия
</button>
```

Требования:

- **CSS‑классы:**
  - `ui-btn` — базовый класс кнопки Bitrix24;
  - один из вариантов оформления:
    - `ui-btn-primary` (по умолчанию);
    - `ui-btn-light-border` и др. — по мере развития модуля;
  - отдельный JS‑селектор:
    - `js-bp-button-field` или аналогичный, стабильный класс для навешивания обработчиков.

- **Обязательные `data-*` атрибуты:**
  - `data-entity-id` — тип сущности CRM (`DEAL`, `LEAD`, `CONTACT`, `COMPANY`, ID смарт‑процесса и т.п.);
  - `data-element-id` — ID элемента CRM (целое число);
  - `data-field-id` — ID пользовательского поля в `b_user_field` (целое число);
  - `data-user-id` — ID текущего пользователя (целое число).

Дополнительно, при необходимости, могут добавляться:
- `data-button-code` — символьный код поля/кнопки;
- `data-settings-id` — ID записи в `my_bpbutton_settings` (если есть).

#### 1.2. Соглашения по атрибутам

- JS‑код **не должен** полагаться на структуру DOM вокруг кнопки (родительские контейнеры), а использовать только:
  - класс‑селектор (`.js-bp-button-field`);
  - `data-*` атрибуты;
  - текст кнопки (при необходимости).
- Любые изменения текста/локализации не должны ломать JS‑логику.

---

### 2. Используемые JS‑модули Bitrix

Для реализации фронтенд‑части рекомендуется использовать следующие модули ядра Bitrix:

- **`ui.buttons`**
  - Базовая стилизация и поведение кнопок (`ui-btn`, `ui-btn-primary`, `ui-btn-wait`, `ui-btn-disabled`).

- **`ui.sidepanel`**
  - Открытие `BX.SidePanel.Instance` с заданным URL, заголовком, шириной и опциями.

- **`ui.notification`**
  - Показ уведомлений пользователю:
    - `BX.UI.Notification.Center.notify({ content: '...', autoHideDelay: 5000 })`.

- **`main.core` / `main.core.events`**
  - Для общего доступа к `BX`, подписки на события, работы с DOM (если требуется).

Подключение модулей может выполняться:
- через `CJSCore::Init` в PHP‑шаблоне;
- через `extension.load` (если используется современная система UI‑расширений Bitrix).

### 3. Жизненный цикл кнопки

#### 3.1. Инициализация

1. После загрузки карточки CRM JS‑код модуля выполняет инициализацию:
   - находит все элементы с классом `.js-bp-button-field`;
   - навешивает обработчик клика (один раз на элемент).
2. При повторной инициализации (например, после частичной перерисовки карточки) код должен:
   - избегать дублирования обработчиков (проверять флаг/атрибут инициализации).

Пример концептуального кода:

```javascript
function initBpButtons() {
  var buttons = document.querySelectorAll('.js-bp-button-field');

  buttons.forEach(function (button) {
    if (button.dataset.bpButtonInited === 'Y') {
      return;
    }

    button.dataset.bpButtonInited = 'Y';
    button.addEventListener('click', handleBpButtonClick);
  });
}
```

#### 3.2. Обработка клика

1. При клике по кнопке:
   - если кнопка уже в состоянии `loading` (`ui-btn-wait`) — игнорировать повторный клик;
   - собрать значения `data-entity-id`, `data-element-id`, `data-field-id`, `data-user-id`.
2. Перевести кнопку в состояние загрузки:
   - добавить класс `ui-btn-wait`;
   - при необходимости установить атрибут/флаг «заблокировано».
3. Отправить запрос к AJAX‑контроллеру (`getConfigAction`), передав:
   - `entityId`, `elementId`, `fieldId`, `sessid`.

#### 3.3. Обработка ответа

- **Успешный ответ (`success: true`):**
  1. Снять с кнопки `ui-btn-wait`, вернуть её в состояние `idle`.
  2. Взять из ответа:
     - `url`, `title`, `width`, `context`.
  3. Вызвать `BX.SidePanel.Instance.open` (или аналогичный метод) с переданными параметрами.

- **Ответ с ошибкой (`success: false`):**
  1. Снять `ui-btn-wait` (кнопка не должна «зависать»).
  2. Прочитать `error.code` и `error.message`.
  3. Показать уведомление пользователю через `BX.UI.Notification.Center.notify`, используя текст `message`.
  4. При необходимости, для внутренних целей, залогировать код/сообщение в консоль/отладчик (без раскрытия лишних деталей).

#### 3.4. Открытие SidePanel

При успешном получении конфигурации:

- Вызов должен учитывать:
  - ширину окна (`width`) — число пикселей или строка с `%`;
  - заголовок (`title`);
  - необходимость передачи контекста через query‑параметры или state (внутреннее соглашение обработчика).

Концептуальный пример:

```javascript
BX.SidePanel.Instance.open(response.data.url, {
  width: response.data.width || '70%',
  title: response.data.title || '',
  cacheable: false
});
```

### 4. Состояния кнопки и визуальные классы

#### 4.1. Сопоставление состояний и классов

- `idle`:
  - классы: `ui-btn ui-btn-primary` (или другой выбранный вид);
  - курсор: `pointer`.

- `loading`:
  - добавляется класс `ui-btn-wait`;
  - повторные клики игнорируются до завершения запроса.

- `disabled`:
  - класс `ui-btn-disabled` или эквивалент;
  - курсор: `default`;
  - обработчик клика либо не навешивается, либо внутри проверки состояние и клик игнорируется.

#### 4.2. Обработка ошибок

- При любой ошибке:
  - состояние `loading` снимается;
  - кнопка возвращается в `idle` (либо, при жёсткой бизнес‑логике, переводится в `disabled` на основе кода ошибки).
- Визуальные эффекты ошибки (подсветка, вибрация и т.п.) описываются в `../ui_ux/crm_card_button.md` и могут быть реализованы поверх базового поведения.

---

### 5. Взаимодействие с другими компонентами интерфейса

- JS‑логика кнопки должна быть максимально изолированной:
  - не вмешиваться в работу других полей/компонентов карточки;
  - не модифицировать DOM вне своей зоны ответственности;
  - не блокировать стандартные действия CRM.

- При использовании нескольких кнопок на одной карточке:
  - каждая работает независимо;
  - общий код инициализации повторно обрабатывает карточку после частичных обновлений интерфейса (например, при смене стадии сделки).

### 6. Чек‑лист для фронтенд‑ревью

1. **Разметка**
   - [ ] User type генерирует кнопку с корректными классами (`ui-btn`, стиль, JS‑селектор).
   - [ ] Присутствуют все обязательные `data-*` атрибуты (`entity`, `element`, `field`, `user`).

2. **JS‑инициализация**
   - [ ] Обработчики кликов навешиваются один раз на каждый элемент.
   - [ ] Повторная инициализация не приводит к дублированию обработчиков.

3. **Состояния кнопки**
   - [ ] При клике кнопка переходит в `loading` (класс `ui-btn-wait`).
   - [ ] Повторные клики во время загрузки игнорируются.
   - [ ] После успешного/ошибочного ответа кнопка возвращается в предыдущее безопасное состояние.

4. **Работа с AJAX‑контроллером**
   - [ ] В запрос передаются все необходимые параметры (`entityId`, `elementId`, `fieldId`, `sessid`).
   - [ ] При `success: true` вызывается `BX.SidePanel` с нужными параметрами.
   - [ ] При `success: false` показывается уведомление пользователю.

5. **Уведомления и UX**
   - [ ] Уведомления выводятся через `BX.UI.Notification.Center`, тексты согласованы с backend‑ошибками.
   - [ ] Нет ситуаций, когда пользователь остаётся без обратной связи при ошибках.

6. **Совместимость**
   - [ ] Кнопка и её поведение корректно выглядят в светлой и тёмной темах Bitrix24.
   - [ ] Несколько кнопок на одной карточке работают независимо друг от друга.
