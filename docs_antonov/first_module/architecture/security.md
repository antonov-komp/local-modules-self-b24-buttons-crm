## Безопасность модуля «BP Button Field»

Документ описывает модель прав, обязательные проверки в backend‑слое (D7), политику ошибок и логирования, а также чек‑лист для security‑ревью.

Ссылается на:
- функциональные/нефункциональные требования (`requirements.md`, NFR‑004 и др.);
- архитектуру backend (`backend_d7.md`);
- спецификацию AJAX‑контроллера (`../api/ajax_controller.md`).

---

### 1. Модель прав и роли

#### 1.1. Основные роли

- **Администратор модуля / системный администратор**
  - Имеет права на:
    - установку/удаление модуля;
    - доступ к административному разделу «Кнопки БП»;
    - редактирование настроек кнопок (`my_bpbutton_settings`);
    - просмотр технических логов модуля.
  - В Bitrix24 определяется как пользователь с правами администратора модулей (`'W'`/`'X'` для модуля `my.bpbutton` или администратор портала).

- **Администратор CRM**
  - Имеет права на:
    - добавление/удаление пользовательских полей в CRM (в т.ч. типа `bp_button_field`);
    - настройку бизнес‑процессов/роботов (в будущем).
  - Не обязательно является администратором модуля, но часто пересекается по ролям.

- **Обычный пользователь CRM (менеджер/оператор)**
  - Может:
    - видеть кнопки в карточках CRM, если имеет права на просмотр соответствующей сущности;
    - нажимать на кнопки;
    - получать доступ к интерфейсам/формам, открываемым через SidePanel (при условии отдельной проверки прав на уровне этих интерфейсов).

- **Аналитик / владелец бизнеса**
  - Как правило, не работает напрямую с модулем, но может:
    - просить предоставлять отчётность по логам;
    - инициировать изменения в конфигурации кнопок.

#### 1.2. Привязка к механизмам прав Bitrix24

- Проверки прав на уровне CRM должны использовать:
  - `\CCrmAuthorizationHelper`, `\CCrmPerms` или D7‑аналоги;
  - стандартные Bitrix‑механизмы определения, может ли пользователь читать/изменять сущность (`DEAL`, `LEAD` и т.п.).
- Проверка прав на административный раздел модуля:
  - через `$APPLICATION->GetGroupRight('my.bpbutton')` в admin‑файлах;
  - пользователи без достаточных прав должны получать отказ доступа до выполнения какой‑либо бизнес‑логики.

---

### 2. Обязательные проверки в backend‑слое

#### 2.1. AJAX‑контроллер (`ButtonController` и др.)

Для всех action‑методов (минимум `getConfigAction`, а также будущих `logClickAction` и т.п.) обязательны следующие шаги:

1. **Проверка сессии**
   - В начале обработки:
     - `if (!check_bitrix_sessid()) { ... INVALID_SESSION ... }`
   - При провале:
     - возвращается JSON‑ошибка с кодом `INVALID_SESSION`;
     - подробности не логируются, так как это нормальная ситуация (истёкшая/отсутствующая сессия).

2. **Проверка прав на сущность CRM**
   - Перед любым доступом к данным CRM:
     - проверяется возможность чтения/записи соответствующей сущности (`entityId`, `elementId`) текущим пользователем;
   - При недостатке прав:
     - возвращается JSON‑ошибка с кодом `ACCESS_DENIED` и безопасным сообщением («Недостаточно прав для доступа к элементу CRM.»);
     - в логах фиксируется факт отказа, но без лишних деталей входных параметров.

3. **Проверка существования и активности настроек**
   - По `fieldId` извлекаются настройки из `my_bpbutton_settings`:
     - при отсутствии записи — `SETTINGS_NOT_FOUND`;
     - при `ACTIVE = 'N'` — `BUTTON_INACTIVE`.
   - В обоих случаях:
     - пользователю возвращается читабельное сообщение;
     - в логах можно зафиксировать диагностическую информацию для администратора.

4. **Валидация входных параметров**
   - Типы:
     - `entityId` — строка с ограничением по длине (набор допустимых значений);
     - `elementId`, `fieldId` — целые положительные числа.
   - При некорректных форматах:
     - возвращается ошибка с общим кодом (например, `BAD_REQUEST` или `INTERNAL_ERROR`, в зависимости от политики проекта);
     - подробности фиксируются только в логах.

5. **Обработка исключений**
   - Внутри контроллера должен быть единый catch‑блок:
     - для предсказуемых бизнес‑ошибок — генерация структурированного ответа с соответствующим кодом;
     - для неожиданных исключений — код `INTERNAL_ERROR` и логирование деталей.

#### 2.2. Административный раздел («Кнопки БП»)

В admin‑файлах модуля (например, `admin/bpbutton_list.php` и форма редактирования) необходимо:

- **Проверять права модуля до выполнения любой логики**
  - Если `$APPLICATION->GetGroupRight('my.bpbutton') == 'D'`:
    - сразу завершать выполнение скрипта, выводя стандартное сообщение о недостатке прав;
    - не выполнять ни запросов к БД, ни рендера интерфейса.

- **Проверять CSRF‑токены для POST‑операций**
  - Для форм сохранения/массовых действий использовать типовой механизм Bitrix (`bitrix_sessid_post()` и проверку `check_bitrix_sessid()`).

- **Ограничивать операции**
  - Пользователь с ограниченными правами (если такие роли предусмотрены) не должен иметь доступа к destructive‑действиям (удаление настроек и т.п.).

#### 2.3. Обработчики событий (user type, OnAfterUserFieldAdd, OnUserFieldDelete)

- Обработчики событий ядра выполняются в доверенном контексте, но должны:
  - минимизировать логику (делегировать в сервисный слой);
  - не выполнять лишних запросов и не раскрывать чувствительные данные в логах.
- При ошибках в обработчиках:
  - по возможности логировать проблему в технические логи модуля;
  - не ломать работу ядра CRM (особенно при удалении полей) — аккуратно обрабатывать исключения.

---

### 3. Политика ошибок

#### 3.1. Формат ошибок в JSON (AJAX‑слой)

Все ошибки, возвращаемые с AJAX‑контроллера, должны иметь единый формат:

```json
{
  "success": false,
  "error": {
    "code": "<КОД_ОШИБКИ>",
    "message": "<Локализованное сообщение для пользователя>"
  }
}
```

- Поле `message`:
  - **всегда локализовано** через `Loc::getMessage`;
  - не содержит внутренних деталей (SQL, стек, имена внутренних классов и т.п.).

#### 3.2. Набор кодов ошибок

Рекомендуемый список кодов (может дополняться):

- `INVALID_SESSION` — не пройдена проверка сессии;
- `ACCESS_DENIED` — недостаточно прав для работы с сущностью;
- `SETTINGS_NOT_FOUND` — отсутствует запись в `my_bpbutton_settings` для указанного `fieldId`;
- `BUTTON_INACTIVE` — кнопка выключена (`ACTIVE = 'N'`);
- `INTERNAL_ERROR` — внутренняя ошибка (любое непредвиденное исключение);
- `BAD_REQUEST` — некорректные/неполные входные параметры (опционально).

Маппинг кодов на тексты сообщений описан в:
- `../api/ajax_controller.md`;
- языковых файлах контроллера.

#### 3.3. Ошибки UI‑слоя

- Frontend, получив ошибку, обязан:
  - вернуть кнопку в безопасное состояние (`idle`, без бесконечного `loading`);
  - показать пользователю уведомление через стандартный UI (`BX.UI.Notification.Center.notify`).
- Все тексты уведомлений должны соответствовать кодам ошибок и быть консистентными.

---

### 4. Политика логирования

#### 4.1. Что логируем

- **Технические ошибки и исключения backend‑слоя**
  - Исключения в контроллере, сервисах, ORM:
    - тип исключения;
    - сообщение;
    - стек (в логах, но не в JSON);
    - контекст (ID сущности, ID поля, пользователь), с учётом маскировки чувствительных данных.

- **Отказы в доступе**
  - Факт `ACCESS_DENIED` с минимумом контекста:
    - пользователь;
    - сущность;
    - элемент.

- **(Расширение) Логи нажатий**
  - Сохраняются в `my_bpbutton_logs`:
    - `SETTINGS_ID`, `FIELD_ID`, `ENTITY_ID`, `ELEMENT_ID`, `USER_ID`;
    - `STATUS`, `MESSAGE`, время.

#### 4.2. Чего не логируем

- Полные запросы/ответы, содержащие персональные данные в открытом виде, без необходимости.
- Полные URL с чувствительными параметрами (`token`, `password` и т.п.) — такие значения должны маскироваться.

#### 4.3. Уровни логов

- **INFO**:
  - успешные ключевые операции (по усмотрению);
  - запуск/остановка модуля.
- **WARNING**:
  - отказ в доступе;
  - проблемы с настройками (отсутствие записей).
- **ERROR**:
  - внутренние исключения;
  - сбои при работе с БД (на уровне модуля).

---

### 5. Чек‑лист безопасности для ревью

При ревью модуля перед релизом рекомендуется пройтись по следующему чек‑листу:

1. **Сессии и CSRF**
   - [ ] Во всех AJAX‑action‑ах вызывается `check_bitrix_sessid()`.
   - [ ] Во всех admin‑формах POST‑запросы защищены сессионным токеном.

2. **Права и доступ**
   - [ ] Доступ к admin‑разделу «Кнопки БП» закрыт для пользователей без прав на модуль.
   - [ ] Перед обращением к данным CRM проверяются права пользователя на сущность/элемент.
   - [ ] При отсутствии прав возвращается код `ACCESS_DENIED` без утечки деталей.

3. **Обработка ошибок**
   - [ ] Все ошибки контроллера возвращаются в структурированном формате JSON.
   - [ ] Пользователь видит только локализованные сообщения, без внутренней информации.
   - [ ] Необработанные исключения перехватываются и маппятся на `INTERNAL_ERROR`.

4. **Логирование**
   - [ ] Критические ошибки и исключения логируются в технические логи модуля.
   - [ ] В логах не появляются сырые чувствительные данные без необходимости.
   - [ ] Логирование отказов в доступе реализовано (минимум для отладки).

5. **Архитектура и разделение ответственности**
   - [ ] Контроллеры не содержат прямых SQL‑запросов и тяжёлой логики — только вызов сервисов.
   - [ ] Обработчики событий ядра минимальны и используют сервисный слой.
   - [ ] Модуль не модифицирует ядро (`/bitrix/modules/`) и ставится только в `/local/modules/`.

6. **UI и сообщения**
   - [ ] Все пользовательские сообщения вынесены в языковые файлы.
   - [ ] Тексты ошибок в UI соответствуют кодам ошибок.
   - [ ] Кнопка никогда не остаётся в «зависшем» состоянии при ошибках.

## Безопасность модуля «BP Button Field»

Документ описывает модель прав и основные меры безопасности для модуля:

- кто может нажимать кнопку;
- кто может настраивать кнопки;
- кто может просматривать логи;
- какие проверки выполняются на уровне backend.

---

### 1. Ограничения по размещению и окружению

- Код модуля размещается **только** в `/local/modules/my.bpbutton/`:
  - исключается вмешательство в ядро `/bitrix/modules/`;
  - упрощает обновление платформы.
- Для работы модуля требуется:
  - актуальная версия ядра Битрикс24 (коробка) с поддержкой D7;
  - включённый модуль `crm`.

---

### 2. Модель прав и роли

#### 2.1. Кто может нажимать кнопку

- Любой пользователь, имеющий права на просмотр соответствующей CRM‑карточки:
  - права определяются стандартными механизмами CRM (`CCrmPerms`, `CCrmAuthorizationHelper`);
  - дополнительных прав на модуль не требуется.
- При вызове AJAX‑контроллера:
  - проверяется сессия (`check_bitrix_sessid()`);
  - проверяется доступ к конкретной сущности (`entityId`/`elementId`) через CRM‑права.

#### 2.2. Кто может настраивать кнопки

- Только администраторы портала или пользователи с правом администратора модулей:
  - доступ к админ‑разделу «Кнопки БП» контролируется:
    - через настройки прав модуля (`$APPLICATION->GetGroupRight('my.bpbutton')`),
    - либо через стандартные проверки `IsAdmin()`.
- В UI реестра:
  - операции добавления/изменения настроек (`HANDLER_URL`, `TITLE`, `WIDTH`, `ACTIVE`) доступны только при достаточных правах;
  - операции удаления настроек также ограничены.

#### 2.3. Кто может видеть логи

- Доступ к логам (таблица `my_bpbutton_logs`) должен быть ограничен:
  - только администраторам или выделенной роли «Аудитор»;
  - просмотр/выгрузка логов — только через специально спроектированные admin‑страницы модуля.

---

### 3. Проверки на уровне контроллера и событий

#### 3.1. Проверка сессии (`check_bitrix_sessid`)

- Каждый AJAX‑запрос к контроллеру модуля должен включать `sessid`:
  - контроллер вызывает `check_bitrix_sessid()` в начале каждого action‑метода;
  - при неуспехе возвращается ошибка безопасности (без выполнения бизнес‑логики).

#### 3.2. Проверка прав на сущность CRM

- При обработке нажатия кнопки:
  - по `entityId` и `elementId` определяется тип сущности (лид, сделка и т.п.);
  - через CRM‑классы (`CCrmPerms`, `\Bitrix\Crm\Security\Controller`) проверяется право на чтение/действие:
    - при отсутствии прав контроллер возвращает ошибку доступа;
    - информация о причине ошибки логируется без раскрытия деталей клиенту.

#### 3.3. Проверка прав на изменение настроек

- В admin‑части:
  - перед сохранением/удалением записей `my_bpbutton_settings` выполняется проверка прав модуля;
  - при отсутствии прав операции блокируются и фиксируются в логах.

#### 3.4. Кастомные роли (потенциал)

- В будущем модуль может определить собственные уровни доступа:
  - например, «Администратор кнопок», «Просмотр логов»;
  - реализация через механизм прав модулей Bitrix (`CMain::GetGroupRight` и настройки прав в админке).

---

### 4. Обработка ошибок и логирование

- Ошибки бизнес‑логики и интеграций:
  - логируются в файл/журнал модуля (через стандартные механизмы Битрикс);
  - сообщения, выводимые пользователю, не содержат технических деталей (stack trace, SQL и т.п.).
- Для логов нажатий (`my_bpbutton_logs`):
  - в поле `MESSAGE` можно хранить только обобщённое диагностическое описание;
  - без персональных данных помимо необходимых идентификаторов (`USER_ID`, `ENTITY_ID`, `ELEMENT_ID`).

---

### 5. Связь с архитектурой backend

- Все проверки, описанные в этом документе, должны:
  - выполняться в контроллере `ButtonController` (см. `backend_d7.md`, `api/ajax_controller.md`);
  - учитываться в admin‑части при работе с `SettingsTable` и `LogsTable`;
  - дополняться тест‑кейсами в разделе `testing/` (проверка сценариев «нет прав», «битая сессия» и т.п.).
