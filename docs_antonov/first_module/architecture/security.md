## Безопасность модуля «BP Button Field»

Документ описывает модель прав и основные меры безопасности для модуля:

- кто может нажимать кнопку;
- кто может настраивать кнопки;
- кто может просматривать логи;
- какие проверки выполняются на уровне backend.

---

### 1. Ограничения по размещению и окружению

- Код модуля размещается **только** в `/local/modules/my.bpbutton/`:
  - исключается вмешательство в ядро `/bitrix/modules/`;
  - упрощает обновление платформы.
- Для работы модуля требуется:
  - актуальная версия ядра Битрикс24 (коробка) с поддержкой D7;
  - включённый модуль `crm`.

---

### 2. Модель прав и роли

#### 2.1. Кто может нажимать кнопку

- Любой пользователь, имеющий права на просмотр соответствующей CRM‑карточки:
  - права определяются стандартными механизмами CRM (`CCrmPerms`, `CCrmAuthorizationHelper`);
  - дополнительных прав на модуль не требуется.
- При вызове AJAX‑контроллера:
  - проверяется сессия (`check_bitrix_sessid()`);
  - проверяется доступ к конкретной сущности (`entityId`/`elementId`) через CRM‑права.

#### 2.2. Кто может настраивать кнопки

- Только администраторы портала или пользователи с правом администратора модулей:
  - доступ к админ‑разделу «Кнопки БП» контролируется:
    - через настройки прав модуля (`$APPLICATION->GetGroupRight('my.bpbutton')`),
    - либо через стандартные проверки `IsAdmin()`.
- В UI реестра:
  - операции добавления/изменения настроек (`HANDLER_URL`, `TITLE`, `WIDTH`, `ACTIVE`) доступны только при достаточных правах;
  - операции удаления настроек также ограничены.

#### 2.3. Кто может видеть логи

- Доступ к логам (таблица `my_bpbutton_logs`) должен быть ограничен:
  - только администраторам или выделенной роли «Аудитор»;
  - просмотр/выгрузка логов — только через специально спроектированные admin‑страницы модуля.

---

### 3. Проверки на уровне контроллера и событий

#### 3.1. Проверка сессии (`check_bitrix_sessid`)

- Каждый AJAX‑запрос к контроллеру модуля должен включать `sessid`:
  - контроллер вызывает `check_bitrix_sessid()` в начале каждого action‑метода;
  - при неуспехе возвращается ошибка безопасности (без выполнения бизнес‑логики).

#### 3.2. Проверка прав на сущность CRM

- При обработке нажатия кнопки:
  - по `entityId` и `elementId` определяется тип сущности (лид, сделка и т.п.);
  - через CRM‑классы (`CCrmPerms`, `\Bitrix\Crm\Security\Controller`) проверяется право на чтение/действие:
    - при отсутствии прав контроллер возвращает ошибку доступа;
    - информация о причине ошибки логируется без раскрытия деталей клиенту.

#### 3.3. Проверка прав на изменение настроек

- В admin‑части:
  - перед сохранением/удалением записей `my_bpbutton_settings` выполняется проверка прав модуля;
  - при отсутствии прав операции блокируются и фиксируются в логах.

#### 3.4. Кастомные роли (потенциал)

- В будущем модуль может определить собственные уровни доступа:
  - например, «Администратор кнопок», «Просмотр логов»;
  - реализация через механизм прав модулей Bitrix (`CMain::GetGroupRight` и настройки прав в админке).

---

### 4. Обработка ошибок и логирование

- Ошибки бизнес‑логики и интеграций:
  - логируются в файл/журнал модуля (через стандартные механизмы Битрикс);
  - сообщения, выводимые пользователю, не содержат технических деталей (stack trace, SQL и т.п.).
- Для логов нажатий (`my_bpbutton_logs`):
  - в поле `MESSAGE` можно хранить только обобщённое диагностическое описание;
  - без персональных данных помимо необходимых идентификаторов (`USER_ID`, `ENTITY_ID`, `ELEMENT_ID`).

---

### 5. Связь с архитектурой backend

- Все проверки, описанные в этом документе, должны:
  - выполняться в контроллере `ButtonController` (см. `backend_d7.md`, `api/ajax_controller.md`);
  - учитываться в admin‑части при работе с `SettingsTable` и `LogsTable`;
  - дополняться тест‑кейсами в разделе `testing/` (проверка сценариев «нет прав», «битая сессия» и т.п.).
