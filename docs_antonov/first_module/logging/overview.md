## Логирование и аналитика модуля «BP Button Field»

Документ описывает общие принципы логирования в модуле, состав и назначение таблицы `my_bpbutton_logs`, а также примеры аналитических запросов.

Ссылается на:
- `../architecture/data_model.md` — формальное описание структуры БД;
- `../tasks/TASK-004-logging.md` — рабочая задача по реализации логирования;
- `../architecture/security.md` — требования по безопасности и логированию.

---

### 1. Что логируем

#### 1.1. События, попадающие в `my_bpbutton_logs`

- Факты нажатий на кнопки `bp_button_field`:
  - успешные (`STATUS = 'SUCCESS'`);
  - завершившиеся с ошибкой (`STATUS = 'ERROR'` и др.);
  - при необходимости — таймауты, отмены и другие статусы.

- Контекст нажатия:
  - какая кнопка (`FIELD_ID` / `SETTINGS_ID`);
  - в какой сущности и элементе CRM (`ENTITY_ID`, `ELEMENT_ID`);
  - кем (`USER_ID`);
  - когда (`CREATED_AT`).

#### 1.2. Что логируем в технические логи (файлы/системные логи)

- Внутренние ошибки и исключения:
  - сбои в работе контроллера;
  - ошибки записи логов;
  - любые неожиданные исключения.

При этом:
- в `my_bpbutton_logs` попадает **деловой факт** (клик, результат, контекст);
- в технические логи — подробная техническая информация (stack trace, SQL, детали исключения).

---

### 2. Чего не логируем

- Чувствительные данные в открытом виде:
  - пароли, токены, секреты интеграций;
  - полные URL с параметрами авторизации.

- Избыточные данные CRM:
  - значения всех полей сделки/лида и т.п.;
  - длинные текстовые комментарии.

При необходимости диагностики:
- допускается хранить укороченные/маскированные значения;
- либо ссылаться на ID сущности и использовать другие механизмы для детализации.

---

### 3. Структура `my_bpbutton_logs` (кратко)

Полное описание приведено в `../architecture/data_model.md`. Краткая выжимка:

- `ID` — первичный ключ.
- `SETTINGS_ID` — ссылка на `my_bpbutton_settings.ID`.
- `FIELD_ID` — ID пользовательского поля (`b_user_field.ID`), дублируется для удобства.
- `ENTITY_ID` — тип сущности CRM (`DEAL`, `LEAD`, `CONTACT`, `COMPANY`, ID смарт‑процесса).
- `ELEMENT_ID` — ID элемента CRM.
- `USER_ID` — ID пользователя, инициировавшего нажатие.
- `STATUS` — строковый статус (`SUCCESS`, `ERROR`, `TIMEOUT` и т.д.).
- `MESSAGE` — текстовое диагностическое сообщение (опционально).
- `CREATED_AT` — дата/время создания записи.

Индексы:
- по `FIELD_ID`;
- по паре `ENTITY_ID` + `ELEMENT_ID`;
- по `USER_ID`.

---

### 4. Сроки хранения логов

Рекомендации (могут быть уточнены полисами компании):

- **Оперативный период** (1–3 месяца):
  - логи используются для быстрой диагностики;
  - активная аналитика по использованию кнопок.

- **Архивный период** (до 12 месяцев и более):
  - логи могут быть:
    - либо оставлены в БД (при разумных объёмах);
    - либо выгружены во внешнее хранилище/архив;
    - либо агрегированы (подведены итоги, а детализация удалена).

При большом объёме данных:
- рекомендуется периодическое архивирование/очистка таблицы;
- может быть реализован отдельный maintenance‑скрипт или задание.

---

### 5. Примеры аналитических запросов

Ниже приведены типовые SQL‑запросы, собранные в одном месте (подробности см. в `data_model.md`).

#### 5.1. Количество нажатий по полям за период

```sql
SELECT FIELD_ID, COUNT(*) AS CNT
FROM my_bpbutton_logs
WHERE CREATED_AT BETWEEN :from AND :to
GROUP BY FIELD_ID
ORDER BY CNT DESC;
```

#### 5.2. Активность пользователей (успешные нажатия)

```sql
SELECT USER_ID, COUNT(*) AS CLICKS
FROM my_bpbutton_logs
WHERE STATUS = 'SUCCESS'
  AND CREATED_AT >= DATE_SUB(NOW(), INTERVAL 30 DAY)
GROUP BY USER_ID
ORDER BY CLICKS DESC;
```

#### 5.3. Подробный журнал по конкретной кнопке

```sql
SELECT *
FROM my_bpbutton_logs
WHERE FIELD_ID = :fieldId
ORDER BY CREATED_AT DESC
LIMIT 100;
```

#### 5.4. Распределение статусов за период

```sql
SELECT STATUS, COUNT(*) AS CNT
FROM my_bpbutton_logs
WHERE CREATED_AT BETWEEN :from AND :to
GROUP BY STATUS;
```

---

### 6. Связь логов с UI

- В админ‑реестре настроек (`bpbutton_list.php`) может быть добавлено действие «Логи»:
  - переход на страницу просмотра логов по конкретной кнопке;
  - фильтры по дате, пользователю, статусу.

- В первой версии допускается отсутствие отдельного UI:
  - аналитика выполняется через SQL и внешние инструменты (BI, Excel);
  - данный документ служит справочником для тех, кто строит отчёты.

---

### 7. Чек‑лист по логированию

1. **Полнота данных**
   - [ ] Каждое нажатие кнопки (успешное) порождает запись в `my_bpbutton_logs`.
   - [ ] Запись содержит `FIELD_ID`, `ENTITY_ID`, `ELEMENT_ID`, `USER_ID`, `STATUS`, `CREATED_AT`.

2. **Безопасность**
   - [ ] В `MESSAGE` и других полях не сохраняются конфиденциальные данные в открытом виде.
   - [ ] Доступ к UI логов (если есть) ограничен правами администраторов.

3. **Производительность**
   - [ ] Индексы соответствуют типичным аналитическим запросам.
   - [ ] Запросы из этого документа выполняются с приемлемым временем отклика на боевых объёмах.
