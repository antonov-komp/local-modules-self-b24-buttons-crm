## Установка и удаление модуля «BP Button Field»

Документ описывает ответственность `install/index.php` модуля:
- какие действия выполняются при установке (`DoInstall`);
- какие действия выполняются при удалении (`DoUninstall`);
- чего модуль **не** делает (границы ответственности).

---

### 1. Общий класс модуля

Расположение:  
`/local/modules/my.bpbutton/install/index.php`

Типовая структура:

- класс `my_bpbutton` (наследник `CModule`);
- свойства:
  - `$MODULE_ID = 'my.bpbutton'`;
  - `$MODULE_VERSION`, `$MODULE_VERSION_DATE` (подтягиваются из `version.php`);
  - `$MODULE_NAME`, `$MODULE_DESCRIPTION` (через `Loc::getMessage`).
- методы:
  - `DoInstall()` — действия при установке;
  - `DoUninstall()` — действия при удалении.

---

### 2. Действия при установке (`DoInstall`)

#### 2.1. Регистрация модуля

- Вызов `RegisterModule('my.bpbutton')`.
- Регистрация автозагрузки классов (через `include.php` / `Loader::registerAutoLoadClasses`).

#### 2.2. Регистрация обработчиков событий

Через `RegisterModuleDependences` (или D7‑аналоги):

- `main`:
  - `OnUserTypeBuildList` → метод в `My\BpButton\EventHandler` для регистрации user type `bp_button_field`.
  - `OnAfterUserFieldAdd` → обработчик auto‑регистрации настроек в `my_bpbutton_settings`.
  - `OnUserFieldDelete` → обработчик очистки зависимых настроек (и логов в будущем).

При необходимости — другие события (чётко описанные в `../api/events.md`).

#### 2.3. Создание таблиц БД

- Проверка наличия таблиц `my_bpbutton_settings` (и, при включении логирования, `my_bpbutton_logs`).
- При отсутствии — создание через:
  - D7‑миграции (предпочтительно);
  - или прямые SQL‑скрипты внутри `DoInstall` (минимальный необходимый набор).

Ключевые моменты:
- не создавать внешних ключей к системным таблицам Bitrix (`b_user_field`, CRM‑таблицы), чтобы не ломать обновления ядра;
- использовать префикс `my_` для избежания конфликтов.

#### 2.4. Установка admin‑прокси

- Копирование файлов:

```text
/local/modules/my.bpbutton/install/admin/my_bpbutton_list.php
/local/modules/my.bpbutton/install/admin/my_bpbutton_edit.php
```

в:

```text
/bitrix/admin/my_bpbutton_list.php
/bitrix/admin/my_bpbutton_edit.php
```

- Сами файлы‑прокси минимальны и просто подключают:
  - `/bitrix/modules/my.bpbutton/admin/bpbutton_list.php`;
  - `/bitrix/modules/my.bpbutton/admin/bpbutton_edit.php`.

#### 2.5. Регистрация пунктов меню

- Подключение `admin/menu.php` модуля, который:
  - добавляет пункт меню «Кнопки БП» в раздел «Настройки → Мои модули»;
  - учитывает права доступа к модулю.

---

### 3. Действия при удалении (`DoUninstall`)

#### 3.1. Снятие регистрации модуля

- Вызов `UnRegisterModule('my.bpbutton')`.
- Снятие автозагрузки классов, если требуется.

#### 3.2. Отписка от событий

- Вызов `UnRegisterModuleDependences` для всех событий, зарегистрированных при установке:
  - `OnUserTypeBuildList`;
  - `OnAfterUserFieldAdd`;
  - `OnUserFieldDelete`;
  - и др. при необходимости.

#### 3.3. Удаление admin‑прокси

- Удаление файлов из `/bitrix/admin/`, созданных при установке:
  - `my_bpbutton_list.php`;
  - `my_bpbutton_edit.php`.

#### 3.4. Обработка таблиц БД

Допускаются два подхода (выбирается проектом и фиксируется в документации):

- **Консервативный:**  
  - таблицы `my_bpbutton_settings` и `my_bpbutton_logs` **не удаляются** при `DoUninstall`;
  - администратору предлагается вручную решить, нужно ли чистить данные;
  - аргумент: история настроек и логов может быть ценна для аудита.

- **Очистка при удалении:**  
  - по запросу администратора (через мастер деинсталляции) модуль может:
    - удалить таблицы с настройками/логами;
    - либо выполнить частичную очистку (например, только настройки, без логов).

В обоих случаях модуль **не должен**:
- изменять/удалять данные CRM (сделки, лиды и т.п.);
- модифицировать системные таблицы ядра.

---

### 4. Чего модуль НЕ делает при install/uninstall

- Не создаёт/не меняет:
  - стандартные таблицы CRM (`b_crm_*`);
  - таблицы пользователей (`b_user`) и др. системные таблицы Bitrix.
- Не добавляет произвольные крон‑задачи/агенты, не описанные в архитектуре.
- Не изменяет системные файлы ядра в `/bitrix/modules/` — всё размещается в `/local/modules/` + admin‑прокси.

---

### 5. Чек‑лист для ревью install/uninstall

1. **Регистрация модуля**
   - [ ] При установке вызывается `RegisterModule('my.bpbutton')`.
   - [ ] При удалении вызывается `UnRegisterModule('my.bpbutton')`.

2. **События**
   - [ ] Все события регистрируются через `RegisterModuleDependences`.
   - [ ] При удалении все зависимости корректно снимаются.

3. **Таблицы БД**
   - [ ] Таблицы создаются только при их отсутствии.
   - [ ] Нет внешних ключей к системным таблицам ядра.
   - [ ] Поведение при удалении таблиц осознанно задокументировано (оставляем/удаляем).

4. **Admin‑прокси**
   - [ ] Файлы‑прокси корректно копируются при установке и удаляются при деинсталляции.
   - [ ] В самих админ‑скриптах используется проверка прав модуля.

5. **Безопасность и чистота ядра**
   - [ ] Модуль не изменяет файлы в `/bitrix/modules/` (кроме прокси в `/bitrix/admin/`).
   - [ ] Все действия установки/удаления обратимы и предсказуемы.

## Установка и удаление модуля «BP Button Field»

Документ описывает:

- логику установки/удаления модуля в `install/index.php`;
- содержимое `version.php` и стратегию версионирования;
- действия при установке: создание таблиц, регистрация событий, добавление пунктов меню;
- действия при удалении: очистка таблиц (опционально), отписка от событий, удаление настроек.

---

### 1. Файл `install/version.php`

Содержит массив с информацией о версии модуля, дате и разработчике. Пример структуры:

```php
$arModuleVersion = [
    'VERSION'      => '0.1.0',
    'VERSION_DATE' => '2026-03-16 00:00:00',
    'NAME'         => 'BP Button Field',
    'DESCRIPTION'  => 'Кнопка управления логикой в карточке CRM',
];
```

Рекомендации по версионированию:

- использовать `MAJOR.MINOR.PATCH`;
- фиксировать изменения в `CHANGELOG` (если ведётся);
- обновлять `VERSION_DATE` при каждой значимой поставке.

---

### 2. Класс модуля в `install/index.php`

Файл `install/index.php` определяет класс модуля (например, `class my_bpbutton extends CModule`) и методы:

- `DoInstall()` — выполняется при установке;
- `DoUninstall()` — выполняется при удалении.

Основные задачи этих методов:

- подключение и регистрация модуля (`RegisterModule`, `CModule::IncludeModule`);
- установка/удаление файлов (admin, компоненты, js/css при необходимости);
- создание/очистка таблиц;
- подписка/отписка на события ядра.

---

### 3. Действия при установке (`DoInstall`)

При установке модуля выполняются следующие шаги:

1. **Регистрация модуля**
   - вызов `RegisterModule('my.bpbutton')`;
   - подключение модуля через `CModule::IncludeModule('my.bpbutton')`.

2. **Создание таблиц БД**
   - на основе D7‑описаний (`SettingsTable`, в будущем `LogsTable`):
     - создание таблицы `my_bpbutton_settings`;
     - (опция будущих версий) создание `my_bpbutton_logs`.
   - рекомендуется использовать `\Bitrix\Main\Application::getConnection()` и `\Bitrix\Main\DB\SqlHelper`.

3. **Регистрация обработчиков событий**
   - событие `main:OnUserTypeBuildList` — регистрация типа `bp_button_field`;
   - событие `main:OnAfterUserFieldAdd` — автоматическое создание записи в `SettingsTable`;
   - событие `main:OnUserFieldDelete` — каскадное удаление настроек и логов.

4. **Установка административных файлов и меню**
   - копирование `admin/bpbutton_list.php` и связанного `menu.php` в нужные директории;
   - регистрация пункта меню «Настройки → Мои модули → Кнопки БП»;
   - настройка прав доступа к модулю (группы и уровни).

5. **Дополнительные шаги (опционально)**
   - инициализация базовых настроек модуля (глобальные опции);
   - создание записей в системных логах о факте установки.

---

### 4. Действия при удалении (`DoUninstall`)

При удалении модуля необходимо аккуратно очистить созданные артефакты:

1. **Отписка от событий**
   - удаление обработчиков `OnUserTypeBuildList`, `OnAfterUserFieldAdd`, `OnUserFieldDelete`;
   - убедиться, что не остаётся «битых» ссылок на классы модуля.

2. **Удаление административных файлов и меню**
   - удаление подключений `bpbutton_list.php` и пункта меню модуля;
   - при необходимости — очистка кеша меню.

3. **Очистка БД (настраиваемое поведение)**
   - в зависимости от бизнес‑решения:
     - либо удалить таблицы `my_bpbutton_settings` и `my_bpbutton_logs`;
     - либо оставить их (для сохранения истории и возможности повторной установки).
   - рекомендуется предоставить администратору выбор (через мастер удаления) — сохранять или удалять данные.

4. **Разрегистрация модуля**
   - вызов `UnRegisterModule('my.bpbutton')`.

---

  - `architecture/backend_d7.md`;
  - `tasks/TASK-001-user-type.md` (регистрация типа и первичная запись в `SettingsTable`);
  - `tasks/TASK-004-logging.md` (подготовка к созданию таблицы логов в будущих версиях).
