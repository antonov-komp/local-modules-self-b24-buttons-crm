## Административный интерфейс и связь с `SettingsTable`

Документ описывает, как административные экраны модуля («Кнопки БП») связаны с ORM‑сущностью `My\BpButton\Internals\SettingsTable` и таблицей `my_bpbutton_settings`.

Ссылается на:
- `../architecture/data_model.md` (описание таблицы);
- `../ui_ux/admin_page.md` (UX‑требования);
- `../tasks/TASK-002-admin-grid.md` (реализация реестра).

---

### 1. Общая схема

- Backend‑слой:
  - ORM‑класс `SettingsTable` (D7) ↔ таблица `my_bpbutton_settings`.
- Admin‑слой:
  - `admin/bpbutton_list.php` — реестр (список);
  - `admin/bpbutton_edit.php` — форма редактирования одной записи.

Связь:

```text
my_bpbutton_settings  <—>  SettingsTable (D7)  <—>  admin/bpbutton_list.php + bpbutton_edit.php
```

---

### 2. Реестр (`bpbutton_list.php`) и `SettingsTable`

#### 2.1. Источник данных

- Список записей для таблицы в `bpbutton_list.php` формируется через:

```php
My\BpButton\Internals\SettingsTable::getList([
    'select' => [...],
    'filter' => [...],
    'order'  => [...],
    'limit'  => ...,
    'offset' => ...
]);
```

- `select` включает:
  - `ID`, `FIELD_ID`, `ENTITY_ID`, `HANDLER_URL`, `TITLE`, `WIDTH`, `ACTIVE`, `UPDATED_AT`;
  - при необходимости — виртуальные поля/связи (например, человеко‑читаемое имя поля из `b_user_field`).

#### 2.2. Фильтры

- UI‑фильтр в admin‑части маппится на `filter` в `getList`:
  - `ENTITY_ID` → `'=ENTITY_ID'`;
  - `ACTIVE` → `'=ACTIVE'`;
  - поиск по названию/коду поля → LIKE‑фильтры по дополнительным полям (через join к `b_user_field` или кэшированное имя);
  - поиск по части URL → `'%HANDLER_URL'`.

#### 2.3. Сортировка и пагинация

- Управляются через параметры `order` и `limit`/`offset`, прокидываемые из `CAdminUiList`/`CAdminList`.

---

### 3. Форма редактирования (`bpbutton_edit.php`) и `SettingsTable`

#### 3.1. Загрузка записи

- По `ID` записи:

```php
$row = SettingsTable::getByPrimary($id)->fetch();
```

- При отсутствии записи:
  - показывается сообщение об ошибке или редирект в список с уведомлением.

#### 3.2. Поля формы и маппинг

Форма редактирования включает:

- `FIELD_ID` / название пользовательского поля:
  - read‑only;
  - может заполняться через join в момент чтения или отдельный запрос к `b_user_field`.

- `ENTITY_ID`:
  - read‑only (или управляемое поле, если поддерживается ограничение по сущности).

- `HANDLER_URL`:
  - редактируемое поле;
  - при сохранении попадает в `HANDLER_URL`.

- `TITLE`:
  - редактируемое поле;
  - сохраняется в `TITLE`.

- `WIDTH`:
  - редактируемое поле;
  - сохраняется в `WIDTH`.

- `ACTIVE`:
  - чекбокс/флаг;
  - сохраняется в `ACTIVE` (`Y`/`N`).

#### 3.3. Сохранение

- При сабмите формы:
  - выполняется проверка сессии (`check_bitrix_sessid()`).
  - собираются значения из полей формы.
  - выполняется валидация:
    - формат URL (`HANDLER_URL`);
    - длина строк (`TITLE`, `WIDTH`);
    - корректность значения `WIDTH` (px/%, базовая проверка).
- При успешной валидации:

```php
SettingsTable::update($id, [
    'HANDLER_URL' => $handlerUrl,
    'TITLE'       => $title,
    'WIDTH'       => $width,
    'ACTIVE'      => $active ? 'Y' : 'N',
]);
```

- При ошибках:
  - форма повторно отображается;
  - ошибки подсвечиваются рядом с соответствующими полями;
  - сообщения локализованы.

---

### 4. Связь с авто‑регистрацией (`OnAfterUserFieldAdd` / `OnUserFieldDelete`)

Админ‑интерфейс должен быть согласован с автоматическими действиями, выполняемыми обработчиками событий:

- `OnAfterUserFieldAdd`:
  - при создании нового поля `bp_button_field` создаётся запись в `my_bpbutton_settings` с:
    - `FIELD_ID` (из события);
    - `ENTITY_ID` (по сущности поля);
    - дефолтными значениями `HANDLER_URL`, `TITLE`, `WIDTH`, `ACTIVE = 'Y'`.
  - эта запись становится видна в реестре без дополнительных действий.

- `OnUserFieldDelete`:
  - при удалении поля типа `bp_button_field` запись в `my_bpbutton_settings` удаляется;
  - при наличии логов — может быть выполнена дополнительная очистка в рамках `TASK-004-logging`.

Таким образом:
- админ‑реестр всегда отображает **только актуальные** пользовательские поля;
- не возникает «битых» записей в админке для уже удалённых полей.

---

### 5. Массовые операции и быстрые действия

#### 5.1. Массовое включение/выключение

- В списке `bpbutton_list.php` должны поддерживаться массовые действия:
  - «Активировать» — установить `ACTIVE = 'Y'` для выбранных записей;
  - «Деактивировать» — установить `ACTIVE = 'N'`.

- Имплементация:
  - сбор списка ID из выбранных строк;
  - передачa в `SettingsTable::update` в цикле или одним батчем (по реализации).

#### 5.2. Удаление настроек

- Допускается опция «Удалить настройки» (с подтверждением), при которой:
  - записи в `my_bpbutton_settings` удаляются;
  - при этом пользовательские поля в CRM **не** удаляются.

- Такое действие должно быть:
  - явно обозначено как потенциально опасное;
  - доступно только администраторам с достаточными правами.

---

### 6. UX‑связь: что видит администратор

С точки зрения UX (`../ui_ux/admin_page.md`):

- `bpbutton_list.php`:
  - отображает «проекцию» таблицы `my_bpbutton_settings` в удобном виде;
  - позволяет быстро найти нужную кнопку по:
    - сущности (`ENTITY_ID`);
    - активности (`ACTIVE`);
    - названию/коду поля;
    - части URL.

- `bpbutton_edit.php`:
  - редактирует одну запись `SettingsTable`;
  - сохраняет изменения обратно в БД через ORM.

---

### 7. Чек‑лист для ревью settings UI

1. **Корректность данных**
   - [ ] Список показывает все записи из `my_bpbutton_settings`, соответствующие user type `bp_button_field`.
   - [ ] Автоматически созданные записи (через `OnAfterUserFieldAdd`) сразу видны в реестре.

2. **Редактирование**
   - [ ] Изменения в `HANDLER_URL`, `TITLE`, `WIDTH`, `ACTIVE` корректно сохраняются в таблицу.
   - [ ] Read‑only поля (`FIELD_ID`, `ENTITY_ID`) не редактируются из формы.

3. **Фильтрация и поиск**
   - [ ] Фильтры по `ENTITY_ID`, `ACTIVE` и названию/коду поля работают ожидаемо.
   - [ ] Поиск по части URL корректно находит записи.

4. **Массовые действия**
   - [ ] Массовое включение/выключение меняет только поле `ACTIVE`.
   - [ ] Потенциально опасные действия (удаление настроек) защищены подтверждением.

5. **Согласованность с backend**
   - [ ] В админ‑скриптах используется `SettingsTable`, а не прямые SQL‑запросы.
   - [ ] Логика не дублирует бизнес‑правила, реализованные в `ButtonService` (минимум дублирования).

## Административный интерфейс настроек «Кнопки БП»

Здесь описывается:

- раздел в админке: «Настройки → Мои модули → Кнопки БП»;
- структура реестра полей: список зарегистрированных `bp_button_field` и их параметры;
- формы редактирования логики: URL обработчика, заголовок окна, ширина SidePanel и др.;
- связь между административным реестром и ORM‑сущностью `SettingsTable`.

---

### 1. Раздел админки и навигация

- Путь: **«Настройки → Мои модули → Кнопки БП»**.
- Источник:
  - файл `admin/menu.php` описывает пункт меню и привязку к правам модуля;
  - при переходе открывается `admin/bpbutton_list.php`.

Раздел является центральной точкой управления всеми полями типа `bp_button_field`.

---

### 2. Реестр полей (список записей `my_bpbutton_settings`)

Административная страница `bpbutton_list.php` отображает данные из `SettingsTable`:

- базовые колонки:
  - `ID` — идентификатор записи настроек;
  - `FIELD_ID` — связь с `b_user_field.ID`;
  - `ENTITY_ID` — (опционально) тип сущности CRM;
  - `HANDLER_URL` — URL обработчика;
  - `TITLE` — заголовок SidePanel;
  - `WIDTH` — ширина SidePanel;
  - `ACTIVE` — флаг активности.

Функции реестра:

- просмотр всех зарегистрированных кнопок;
- фильтрация по `ENTITY_ID`, `ACTIVE`, частичному совпадению `HANDLER_URL` и др.;
- переход к форме редактирования настроек конкретной кнопки.

---

### 3. Формы редактирования логики кнопки

Для каждой записи `my_bpbutton_settings` доступна форма редактирования, которая позволяет:

- задать/изменить:
  - **URL обработчика** (`HANDLER_URL`);
  - **заголовок окна** (`TITLE`);
  - **ширину SidePanel** (`WIDTH`);
  - **активность** (`ACTIVE`);
- просмотреть связанный `FIELD_ID` и (опционально) `ENTITY_ID` в режиме только чтения;
- сохранить изменения с валидацией:
  - проверка формата URL;
  - проверка допустимых значений ширины;
  - обязательность критичных полей (по требованиям бизнеса).

Связь с `SettingsTable`:

- формы используют D7‑ORM (`SettingsTable`) для загрузки и сохранения данных;
- все изменения отражаются в таблице `my_bpbutton_settings` и становятся доступны `ButtonService`.

---

### 4. Связь с `TASK-002-admin-grid.md`

Документ `tasks/TASK-002-admin-grid.md` описывает:

- требования к административному реестру;
- сценарии авто‑регистрации новых полей;
- критерии приёмки.

Настоящий файл (`settings_ui.md`) фиксирует архитектурный уровень:

- какие таблицы и ORM‑классы используются (`SettingsTable`);

UI‑детали (конкретные колонки, фильтры, элементы управления) согласуются с документом `ui_ux/admin_page.md`.
