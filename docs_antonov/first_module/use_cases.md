## Use Cases: сценарии использования модуля «BP Button Field»

Документ описывает ключевые пользовательские сценарии (user journey) для разных ролей:
- администратор CRM;
- менеджер/оператор;
- бизнес‑владелец/аналитик.

Каждый сценарий может ссылаться на требования из `requirements.md` (FR‑xxx, NFR‑xxx).

---

### UC-001 — Создание кнопки для запуска кастомной формы в сделке

**Роль:** Администратор CRM  
**Цель:** Добавить в карточку сделки кнопку, которая открывает внутреннюю форму (iframe в SidePanel) для заполнения дополнительных данных.

**Шаги:**
1. Администратор открывает настройки CRM → Пользовательские поля → Сущность «Сделка».
2. Создаёт новое пользовательское поле с типом `bp_button_field`.
3. Переходит в административный раздел «Настройки → Мои модули → Кнопки БП».
4. Находит созданную кнопку в реестре (по сущности/названию поля).
5. Открывает карточку настройки кнопки и указывает:
   - `HANDLER_URL` — путь до внутреннего обработчика (например, `/local/bpbutton/handlers/deal_form.php`);
   - `TITLE` — «Форма доп. параметров сделки»;
   - `WIDTH` — `70%`;
   - `ACTIVE` — `Y`.
6. Сохраняет изменения и убеждается, что запись в реестре обновлена.

**Ожидаемый результат:**
- В карточке сделки отображается кнопка с корректным названием и стилем.
- При клике открывается SidePanel с указанной формой, без перезагрузки карточки.

**Основные требования:** FR‑001, FR‑002, FR‑004, FR‑005, FR‑006, FR‑008.

### UC-002 — Использование кнопки менеджером в ежедневной работе

**Роль:** Менеджер по продажам / оператор CRM  
**Цель:** Быстро открыть дополнительный интерфейс (форму/отчёт) по конкретной сделке, не покидая карточку.

**Шаги:**
1. Менеджер открывает карточку сделки в CRM.
2. Видит в области пользовательских полей кнопку (например, «Доп. параметры»).
3. Нажимает кнопку:
   - кнопка визуально переходит в состояние загрузки (`loading`);
   - повторные клики временно игнорируются.
4. Через 1–2 секунды открывается SidePanel с нужным интерфейсом в контексте текущей сделки.
5. После открытия окна кнопка возвращается в состояние `idle`.

**Ожидаемый результат:**
- Переход в состояние загрузки происходит мгновенно.
- Окно SidePanel открывается с корректным заголовком и шириной.
- В обработчик передаются корректные параметры (ID сделки, пользователь и т.д.).

**Основные требования:** FR‑002, FR‑003, FR‑008, NFR‑002, NFR‑004.

### UC-003 — Обработка ошибки из‑за отсутствия настроек

**Роль:** Менеджер по продажам / оператор CRM  
**Цель:** Понять, что кнопка не настроена, и не застрять в непонятном состоянии UI.

**Предпосылка:**  
Поле `bp_button_field` создано администратором, но настройки в `my_bpbutton_settings` отсутствуют или неполные (например, нет `HANDLER_URL`).

**Шаги:**
1. Менеджер открывает карточку CRM и видит кнопку.
2. Нажимает кнопку.
3. AJAX‑контроллер возвращает ошибку `SETTINGS_NOT_FOUND` или эквивалент.
4. Кнопка возвращается из `loading` в `idle`.
5. На экране показывается уведомление:
   - «Кнопка не настроена. Обратитесь к администратору.»

**Ожидаемый результат:**
- Интерфейс не зависает, кнопка снова кликабельна (или остаётся в нормальном состоянии).
- Менеджер получает понятное сообщение без технических деталей.

**Основные требования:** FR‑003, FR‑007, FR‑009, NFR‑004, NFR‑005.

### UC-004 — Администратор массово отключает кнопку на время эксперимента

**Роль:** Администратор CRM  
**Цель:** Временно отключить кнопку(и) без удаления пользовательских полей в CRM, чтобы не ломать отчёты и интеграции.

**Шаги:**
1. Администратор открывает «Настройки → Мои модули → Кнопки БП».
2. Находит одну или несколько записей, соответствующих нужным кнопкам (по сущности/названию/URL).
3. Через массовую операцию или форму редактирования меняет `ACTIVE` на `N`.
4. Сохраняет изменения.
5. Просит менеджера открыть карточку CRM и проверить кнопку.

**Ожидаемый результат:**
- В карточке:
  - либо кнопка отображается в неактивном состоянии;
  - либо при клике пользователь получает сообщение о недоступности действия.
- Восстановление работы кнопки сводится к повторному включению `ACTIVE = Y` без удаления полей.

**Основные требования:** FR‑004, FR‑005, FR‑006, FR‑009, NFR‑006.

### UC-005 — Аналитик смотрит статистику по нажатиям (расширенная версия)

**Роль:** Бизнес‑владелец / аналитик (через технического специалиста или SQL/отчёты)  
**Цель:** Понять, насколько активно используется конкретная кнопка в разных сущностях и пользователями.

**Предпосылка:**  
Реализовано логирование в `my_bpbutton_logs` (FR‑010).

**Шаги (базовый сценарий без UI‑отчётов):**
1. Технический специалист или аналитик формирует запросы к БД по `my_bpbutton_logs`:
   - количество нажатий по `FIELD_ID` за период;
   - активность пользователей (`USER_ID`);
   - распределение статусов (`SUCCESS`/`ERROR`).
2. На основе выгрузки строятся отчёты во внешнем инструменте (BI, Excel и т.д.).

**Ожидаемый результат:**
- Логи содержат достаточно данных для аналитики:
  - привязка к полю, сущности, элементу, пользователю, времени и статусу;
  - при необходимости — диагностическое сообщение.
- Запросы выполняются с приемлемой скоростью на рабочих объёмах.

**Основные требования:** FR‑010, NFR‑001, NFR‑003.

### UC-006 — Тестировщик проверяет модуль перед релизом

**Роль:** Тестировщик / QA  
**Цель:** Убедиться, что все ключевые сценарии работы кнопки и админ‑реестра выполняются без регрессий.

**Шаги:**
1. По `requirements.md` и данному документу формирует список тест‑кейсов:
   - создание полей разных сущностей (UC‑001);
   - ежедневное использование кнопки (UC‑002);
   - ошибки настроек/прав (UC‑003);
   - управление активностью кнопок (UC‑004);
   - (при наличии) логирование и простейшая аналитика (UC‑005).
2. Последовательно выполняет шаги каждого use case на стенде.
3. Фиксирует результаты в тестовой документации:
   - фактическое поведение;
   - соответствие/несоответствие требованиям.

**Ожидаемый результат:**
- Картина покрытия тестами привязана к требованиям FR/NFR.
- При появлении новых фич легко добавить новые use cases по аналогии.

**Основные требования:** все FR/NFR на уровне интеграционного поведения, раздел «Тестирование» в TASK‑файлах.
