## Модуль «BP Button Field» для 1С‑Битрикс24 (Коробка)

Этот документ является **master‑спецификацией** модуля «BP Button Field» и связывает бизнес‑требования, реализуемые фичи и технические артефакты (TASK‑файлы, архитектуру, UI, API, логирование).

---

### 1. Цель проекта

Создать программный модуль, расширяющий стандартные возможности CRM Битрикс24 (коробка) путём добавления нового типа пользовательского поля — **«Кнопка управления логикой»** (`bp_button_field`).

Модуль должен позволять:

- администратору **создавать функциональные кнопки** в карточках CRM и смарт‑процессов;
- при нажатии кнопки **открывать интерфейсы** (iframe/`BX.SidePanel`) или запускать бизнес‑логику **без перезагрузки страницы**;
- централизованно **управлять поведением кнопок** через административный реестр;
- в будущих версиях — **логировать нажатия** и интегрироваться с бизнес‑процессами/роботами.

Ограничение: целевая платформа — **коробочная редакция Битрикс24**.

---

### 2. Сводка бизнес‑требований

#### 2.1. Для администратора CRM

- Возможность добавить новое поле типа `bp_button_field`:
  - во всех ключевых сущностях CRM: **Лиды, Сделки, Контакты, Компании, Смарт‑процессы**;
  - через стандартный интерфейс «Добавить поле».
- Доступ к административному реестру модуля:
  - просмотр всех зарегистрированных кнопок;
  - настройка поведения каждой кнопки (куда ведёт, как отображается, какие параметры передаются).

#### 2.2. Для пользователя CRM (менеджера/оператора)

- В карточке CRM вместо текстового поля видеть **интуитивную кнопку**:
  - корректный стиль в соответствии с Bitrix24 UI Kit;
  - понятное название и состояние загрузки.
- При нажатии:
  - моментальная реакция (состояние `state-wait`, защита от даблкликов);
  - открытие SidePanel/iframe без перезагрузки карточки;
  - доступ к дополнительным сценариям (формы, отчёты, внешние сервисы) в контексте конкретной сущности.

#### 2.3. Для бизнес‑владельца/аналитика

- Возможность расширять процессы CRM без разработки отдельных модулей под каждый сценарий:
  - новые кнопки могут вести на разные интерфейсы/сервисы при сохранении единого UX;
  - в перспективе — **аудит** (отслеживание факта нажатий, результатов и ошибок).

### 3. Реализуемые фичи (MVP)

#### 3.1. Регистрация кастомного типа поля (User Type)

- Идентификатор типа: `bp_button_field`.
- Доступность:
  - тип должен быть виден и доступен при создании пользовательских полей во всех CRM‑сущностях и смарт‑процессах.
- Параметры поля (на стороне CRM):
  - **Название** — заголовок поля в системе;
  - **Технический ID** — уникальный код (символьный код) для идентификации кнопки в логике модуля;
  - **Визуал** — выбор стиля кнопки (набор предустановленных вариантов на базе Bitrix UI Kit, например `ui-btn-primary`, `ui-btn-light-border`).

> Подробности реализации: см. `tasks/TASK-001-user-type.md`.

#### 3.2. Поведение в интерфейсе CRM (View & Action)

- Отображение в режиме просмотра:
  - вместо текстового значения поля выводится **активная кнопка**;
  - кнопка содержит необходимые `data-*` атрибуты (`entity`, `element`, `field`, `user`).
- Механика клика:
  - переход в состояние «ожидание» (`state-wait`), блокировка повторных нажатий;
  - вызов JS‑расширения модуля, которое запрашивает конфигурацию у AJAX‑контроллера.
- Действие:
  - открытие `BX.SidePanel.Instance` (слайдер) с загрузкой контента через iframe;
  - в обработчик/iframe передаются параметры:
    - `ENTITY_ID` (тип сущности),
    - `ELEMENT_ID` (ID записи),
    - `FIELD_ID` (ID конкретного поля),
    - `USER_ID` (кто нажал).

> UX‑аспекты описываются в `ui_ux/crm_card_button.md`.

#### 3.3. Административный реестр полей (Конструктор)

- Размещение: административный раздел Битрикс24  
  `Настройки → Мои модули → Кнопки БП`.
- Авто‑регистрация:
  - при создании в CRM поля типа `bp_button_field` запись автоматически попадает в реестр (таблица `my_bpbutton_settings`);
  - при удалении поля настройки очищаются.
- Управление логикой:
  - **URL обработчика** — ссылка на internal/внешний обработчик, открываемый в iframe;
  - **Заголовок окна** — текст в шапке SidePanel;
  - **Ширина окна** — ширина SidePanel (px/%, пресеты).

> Поведение и UI реестра описываются в `tasks/TASK-002-admin-grid.md` и `ui_ux/admin_page.md`.

### 4. Техническая архитектура (Backend D7 — краткое резюме)

Полное описание находится в `architecture/backend_d7.md` и `architecture/data_model.md`. Ниже — только связка с фичами.

#### 4.1. Структура хранения данных

- Для управления логикой кнопок используется собственная ORM‑сущность:
  - **Таблица**: `my_bpbutton_settings`;
  - **Класс**: `My\BpButton\Internals\SettingsTable` (наследник `DataManager`);
  - **Связи**: поле `FIELD_ID` связывает запись с системной таблицей `b_user_field`.

#### 4.2. Обработка событий (Events)

- `main:OnUserTypeBuildList` — регистрация типа поля `bp_button_field` при инициализации модуля.
- `main:OnAfterUserFieldAdd` — автоматическое создание записи в `SettingsTable` при добавлении нового поля в CRM.
- `main:OnUserFieldDelete` — каскадная очистка настроек модуля при удалении поля.

> Подробные сценарии — в `api/events.md`, а конкретные шаги реализации — в `tasks/TASK-001-user-type.md` и `tasks/TASK-002-admin-grid.md`.

#### 4.3. AJAX‑контроллер

- Собственный контроллер на базе `Bitrix\Main\Engine\Controller`:
  - безопасно принимает запросы от кнопок;
  - проверяет сессию (`check_bitrix_sessid()`) и права (`CCrmAuthorizationHelper`/`CCrmPerms`);
  - возвращает конфигурацию для `BX.SidePanel` и, в перспективе, иные данные.

> Спецификация методов и форматов — в `api/ajax_controller.md`, а реализационные шаги — в `tasks/TASK-003-ajax-api.md`.

### 5. Стек технологий и стандарты

- **Язык и платформа**: PHP 8.x, D7‑подход, коробочная версия Битрикс24.
- **Frontend**:
  - Bitrix Core JS (`BX`),
  - `ui.buttons`, `ui.sidepanel`, `ui.notification` и связанные компоненты.
- **Локализация**:
  - обязательное использование языковых файлов через `Loc::getMessage`.
- **Безопасность**:
  - размещение только в `/local/modules/`;
  - проверка сессии через `check_bitrix_sessid()`;
  - проверка прав перед любым действием, затрагивающим CRM‑данные;
  - следование стандартам PSR‑12 и гайдам Битрикс.

### 6. План расширения (будущие версии)

- **Условия отображения**:
  - зависимость видимости и доступности кнопки от стадии сделки, значений других полей или ролей пользователя.
- **Интеграция с Роботами/БП**:
  - возможность имитировать «нажатие» кнопки из робота бизнес‑процесса;
  - сценарии автоматического открытия интерфейсов без участия пользователя.
- **Логирование**:
  - таблица `my_bpbutton_logs` для хранения истории всех нажатий;
  - фиксация контекста (пользователь, сущность, поле, результат, время);
  - аналитические сценарии (отчёты по использованию кнопок).

> Концепция логирования описана здесь, технические детали и реализация — в `tasks/TASK-004-logging.md` и `architecture/data_model.md`.
