## Требования к модулю «BP Button Field»

Документ фиксирует формальные требования к модулю для коробочной версии 1С‑Битрикс24 и служит опорой для TASK‑файлов и тест‑плана.

ID требований используются в:
- TASK‑файлах (`TASK-001–TASK-00X`);
- документах архитектуры;
- тестовой документации (`testing/test_plan.md`, `testing/cases.md`).

---

### 1. Общие допущения и контекст

- Платформа: **1С‑Битрикс24, коробочная редакция**.
- Размещение модуля: `/local/modules/my.bpbutton/`.
- Целевая область: CRM‑сущности и смарт‑процессы.
- Пользовательский тип поля: `bp_button_field`.

---

### 2. Функциональные требования (FR)

#### FR-001 — Регистрация типа пользовательского поля

- Тип `bp_button_field` должен быть доступен в стандартном интерфейсе создания пользовательских полей CRM.
- Поддерживаемые сущности:
  - Лид;
  - Сделка;
  - Контакт;
  - Компания;
  - Смарт‑процессы (универсально, без жёсткой привязки к конкретному типу).

Связанные артефакты:
- `TASK-001-user-type.md`
- `architecture/backend_d7.md`

#### FR-002 — Отображение поля как кнопки в карточке CRM

- При просмотре карточки CRM вместо текстового значения поля типа `bp_button_field` должна отображаться интерактивная кнопка.
- Кнопка использует стили Bitrix24 UI Kit (минимум `ui-btn` + предопределённый вариант оформления).
- Кнопка содержит `data-*` атрибуты:
  - `data-entity-id` — тип сущности;
  - `data-element-id` — ID элемента;
  - `data-field-id` — ID пользовательского поля;
  - `data-user-id` — ID текущего пользователя.

Связанные артефакты:
- `TASK-001-user-type.md`
- `ui_ux/crm_card_button.md`

#### FR-003 — Состояния кнопки и поведение при клике

- Кнопка должна поддерживать минимум состояния:
  - `idle` (активна, без загрузки);
  - `loading` (после клика — визуальный индикатор ожидания, блокировка повторного нажатия);
  - `disabled` (недоступна по настройкам/правам).
- При клике:
  - кнопка переходит в состояние `loading`;
  - отправляется запрос к AJAX‑контроллеру (`getConfigAction`);
  - по результату запроса кнопка возвращается в `idle` или в визуальное состояние ошибки (с уведомлением пользователю).

Связанные артефакты:
- `ui_ux/crm_card_button.md`
- `api/ajax_controller.md`

#### FR-004 — Административный реестр кнопок

- В административном разделе должен быть доступен список всех кнопок (`my_bpbutton_settings`), доступный по пути:
  - «Настройки → Мои модули → Кнопки БП».
- В списке отображаются как минимум:
  - ID настройки;
  - поле (человеко‑читаемое название + код);
  - сущность (`ENTITY_ID`);
  - `HANDLER_URL`;
  - `TITLE`;
  - `WIDTH`;
  - `ACTIVE`;
  - дата изменения.
- Должны быть доступны фильтры:
  - по сущности;
  - по активности;
  - по части названия/кода поля;
  - по части URL.

Связанные артефакты:
- `TASK-002-admin-grid.md`
- `ui_ux/admin_page.md`

#### FR-005 — Редактирование настроек кнопки

- Из реестра должна открываться форма редактирования настройки, позволяющая изменить:
  - `HANDLER_URL`;
  - `TITLE`;
  - `WIDTH`;
  - `ACTIVE`.
- `FIELD_ID` и сущность отображаются в read‑only режиме.
- При успешном сохранении пользователь получает уведомление в стиле Bitrix24.

Связанные артефакты:
- `TASK-002-admin-grid.md`
- `module_structure/settings_ui.md` (план)

#### FR-006 — Автоматическая регистрация и очистка настроек

- При создании пользовательского поля типа `bp_button_field` в CRM:
  - должна автоматически создаваться запись в `my_bpbutton_settings` с базовыми значениями.
- При удалении поля:
  - запись в `my_bpbutton_settings` должна быть удалена;
  - при наличии логов — должна быть предусмотрена стратегия их обработки (см. FR‑010).

Связанные артефакты:
- `TASK-001-user-type.md`
- `architecture/backend_d7.md`
- `api/events.md` (план)

#### FR-007 — AJAX‑получение конфигурации кнопки

- Должен быть реализован контроллер `ButtonController` (или аналог) на базе `Bitrix\Main\Engine\Controller` с методом:
  - `getConfigAction(string $entityId, int $elementId, int $fieldId): array`.
- При корректных параметрах и наличии прав метод возвращает JSON:
  - `url` — URL обработчика для SidePanel/iframe;
  - `title` — заголовок окна;
  - `width` — ширина;
  - `context` — объект с информацией о сущности/элементе/поле/пользователе.

Связанные артефакты:
- `TASK-003-ajax-api.md`
- `api/ajax_controller.md`

#### FR-008 — Открытие SidePanel/iframe

- На основе ответа `getConfigAction` фронтенд должен:
  - открыть `BX.SidePanel` с заданным URL/шириной/заголовком;
  - передать в URL параметры: `ENTITY_ID`, `ELEMENT_ID`, `FIELD_ID`, `USER_ID` (или эквивалентные).

Связанные артефакты:
- `ui_ux/crm_card_button.md`
- `architecture/frontend_ui.md` (план)

#### FR-009 — Обработка ошибок на уровне UX

- При типовых ошибках (нет прав, нет настроек, кнопка выключена, внутренняя ошибка) пользователь должен:
  - вернуть кнопку в нормальное состояние (без зависания в `loading`);
  - увидеть краткое и понятное уведомление без технических деталей.
- Ошибки должны иметь коды (`INVALID_SESSION`, `ACCESS_DENIED`, `SETTINGS_NOT_FOUND`, `BUTTON_INACTIVE`, `INTERNAL_ERROR`), маппящиеся на локализованные сообщения.

Связанные артефакты:
- `api/ajax_controller.md`
- `ui_ux/crm_card_button.md`

#### FR-010 — Логирование нажатий (расширение)

- Модуль должен поддерживать концепцию логирования в таблицу `my_bpbutton_logs`:
  - фиксировать факт нажатия;
  - контекст (пользователь, сущность, поле, время);
  - статус (`SUCCESS`, `ERROR`, `TIMEOUT` и т.п.);
  - диагностическое сообщение (опционально).
- Наличие логирования может быть отнесено ко второй версии модуля, но архитектура первой версии не должна ему противоречить.

Связанные артефакты:
- `architecture/data_model.md`
- `TASK-004-logging.md` (план)

---

### 3. Нефункциональные требования (NFR)

#### NFR-001 — Совместимость и поддерживаемые версии

- Модуль должен работать на поддерживаемых версиях коробочного Битрикс24 с:
  - D7‑ядром;
  - PHP версии не ниже 8.0 (конкретная минимальная версия фиксируется в `install/index.php`).
- Модуль не должен требовать модификации ядра в `/bitrix/modules/` и должен устанавливаться только в `/local/modules/`.

#### NFR-002 — Производительность и отзывчивость

- Переход кнопки в состояние `loading` — не более 100 мс с момента клика.
- Открытие SidePanel при нормальном соединении — целевое время 1–2 секунды.
- Запросы к `getConfigAction` должны быть:
  - по возможности лёгкими (использовать индексы и простые выборки к `my_bpbutton_settings`);
  - не блокировать основной интерфейс CRM.

#### NFR-003 — Масштабируемость и независимость от бизнес‑логики обработчиков

- Модуль не должен накладывать жёстких ограничений на реализацию обработчиков, на которые ведут кнопки (внутренние/внешние URL).
- Смена логики обработчика должна производиться через смену `HANDLER_URL` в настройках, без правок кода модуля.

#### NFR-004 — Безопасность

- Все вызовы AJAX‑контроллера:
  - проходят проверку сессии (`check_bitrix_sessid()`);
  - проходят проверку прав на соответствующую CRM‑сущность.
- Внешние URL в `HANDLER_URL` не должны исполняться внутри модуля — модуль только передаёт их в SidePanel/iframe.
- В ответах API не должны раскрываться внутренние детали (стек, SQL и т.п.) — только коды ошибок и человеко‑читаемые сообщения.

Связанные артефакты:
- `architecture/security.md` (план)

#### NFR-005 — Локализация и UI‑стандарты

- Все пользовательские тексты должны быть вынесены в языковые файлы и использовать `Loc::getMessage`.
- Используемые классы UI должны соответствовать Bitrix24 UI Kit.
- Интерфейсы (кнопка, реестр, формы) должны корректно отображаться в светлой и тёмной темах Bitrix24.

#### NFR-006 — Поддерживаемость и расширяемость

- Архитектура модуля должна разделять:
  - ORM‑слой (таблицы настроек/логов);
  - сервисный слой (бизнес‑логика кнопок);
  - слой интеграции с ядром (user type, события);
  - HTTP/AJAX‑слой (контроллеры).
- Добавление новых фич (условия отображения, интеграция с БП, дополнительные параметры) должно происходить без изменения уже опубликованного контракта API, по возможности через расширение.

---

### 4. Ограничения и out of scope

#### OOS-001 — Облачная версия Битрикс24

- Модуль проектируется **только** для коробочной редакции Битрикс24.
- Поддержка облачной версии (REST‑встраивания) находится вне рамок текущего модуля.

#### OOS-002 — Реализация бизнес‑логики обработчиков

- Модуль **не отвечает** за реализацию бизнес‑логики в обработчиках, на которые ведёт кнопка:
  - формы, отчёты, внешние сервисы, интеграции реализуются отдельно;
  - модуль предоставляет только универсальный механизм кнопки и передачи контекста.

#### OOS-003 — Глубокая кастомизация UI карточек CRM

- Модуль не изменяет кардинально макет карточек CRM:
  - не добавляет свои вкладки/блоки, кроме использования существующего механизма пользовательских полей и стандартных размещений.

#### OOS-004 — Полноценный отчётный модуль по логам

- В рамках первой версии не предполагается полноценный UI‑модуль аналитики логов (диаграммы, дашборды).
- Возможны только базовые SQL‑запросы и отчётность средствами БД/внешних инструментов.

## Требования к модулю «BP Button Field»

Модуль предназначен для коробочной версии 1С‑Битрикс24 и реализуется как отдельный модуль в `/local/modules/`.

---

### 1. Общее описание

**Цель проекта:**  
Создать программный модуль, расширяющий стандартные возможности CRM Битрикс24 за счёт добавления нового типа пользовательского поля — «Кнопка управления логикой» (`bp_button_field`).  

**Ключевая идея:**  
Администратор может добавлять в карточки CRM (лиды, сделки, контакты, компании, смарт‑процессы) кнопки, которые:

- выглядят и ведут себя как стандартные элементы Bitrix24 UI Kit;
- по клику открывают интерфейс во `BX.SidePanel` / iframe **без перезагрузки страницы**;
- передают в обработчик контекст CRM‑элемента;
- могут в дальнейшем использоваться в бизнес‑процессах/роботах для автоматизации.

---

### 2. Функциональные требования

#### 2.1. Регистрация пользовательского типа поля

- **Тип поля:** `bp_button_field` (user type).
- **Регистрация:** через событие `main:OnUserTypeBuildList` при инициализации модуля.
- **Доступность:**
  - поле доступно к созданию во всех основных сущностях CRM:
    - Лиды;
    - Сделки;
    - Контакты;
    - Компании;
    - Смарт‑процессы.
  - добавление через стандартный интерфейс «Добавить поле» в настройках CRM.
- **Параметры при создании поля:**
  - **Название** — заголовок поля в системе (отображается в списках полей);
  - **Технический ID** — уникальный код кнопки для идентификации в логике модуля;
  - **Визуальный стиль кнопки** — выбор из преднастроенных вариантов Bitrix UI Kit
    (например, `ui-btn-primary`, `ui-btn-light-border` и др.).

#### 2.2. Поведение поля в интерфейсе CRM

- **Режим отображения карточки (view):**
  - вместо обычного текстового значения поля на форме выводится **активная кнопка**;
  - значение поля как таковое в интерфейсе не редактируется пользователем.
- **Состояния кнопки:**
  - **Нормальное состояние** — доступна для нажатия;
  - **state-wait / лоадер** — после клика:
    - визуально показывает выполнение действия;
    - блокирует повторные нажатия до получения ответа.
- **Действие при клике:**
  - вызывается JS‑расширение модуля (на базе Bitrix Core JS);
  - расширение запрашивает у backend‑контроллера конфигурацию для `BX.SidePanel`;
  - открывается `BX.SidePanel.Instance` с загрузкой контента через iframe.
- **Передаваемый контекст в обработчик (iframe/URL):**
  - `ENTITY_ID` — тип сущности (лид, сделка, контакт, компания, смарт‑процесс);
  - `ELEMENT_ID` — ID конкретной записи CRM;
  - `FIELD_ID` — ID пользовательского поля (связка с настройками модуля);
  - `USER_ID` — ID пользователя, который нажал кнопку.

#### 2.3. Административный реестр полей (конструктор логики)

- **Раздел в админке:**
  - путь: «Настройки → Мои модули → Кнопки БП»;
  - раздел показывает реестр всех полей типа `bp_button_field`.
- **Автоматическая регистрация:**
  - при создании пользовательского поля типа `bp_button_field` в CRM:
    - модуль автоматически создаёт соответствующую запись в своей таблице настроек
      (`my_bpbutton_settings`) через событие `OnAfterUserFieldAdd`.
- **Управление логикой для каждого поля:**
  - **URL обработчика** — ссылка на файл в коробке или внешний сервис, который будет открыт в iframe;
  - **Заголовок окна** — строка, отображаемая в шапке слайдера `SidePanel`;
  - **Ширина окна** — конфигурируемое значение (px или %, в разумных пределах).
- **Удаление поля:**
  - при удалении пользовательского поля `bp_button_field`:
    - модуль должен каскадно удалить связанные настройки из `my_bpbutton_settings`
      (через событие `OnUserFieldDelete`).

---

### 3. Нефункциональные требования

#### 3.1. Производительность и UX

- Открытие `SidePanel` по нажатию кнопки должно происходить **без перезагрузки страницы**.
- Время реакции интерфейса (появление state‑wait, запуск запроса) — не более 200–300 мс.
- Время загрузки контента в `SidePanel` зависит от внешнего обработчика, но:
  - модуль должен предоставлять стабильный UX (лоадер/индикатор);
  - в случае ошибки пользователь получает информативное сообщение.

#### 3.2. Совместимость и окружение

- **Платформа:** коробочная версия 1С‑Битрикс24 (актуальная версия ядра).
- **PHP:** 8.x, с использованием D7‑подхода (ORM, `DataManager`, Engine‑контроллеры).
- **Frontend:**
  - Bitrix Core JS (`BX`), библиотеки:
    - `ui.buttons`;
    - `ui.sidepanel`;
    - `ui.notification`.
- **Модульность:**
  - реализация только в директории `/local/modules/` (без размещения в `/bitrix/modules/`);
  - возможность установки/удаления через стандартный механизм модулей.

#### 3.3. Локализация

- Все пользовательские и административные тексты должны быть вынесены в языковые файлы и
  получаться через `\Bitrix\Main\Localization\Loc::getMessage()`.
- Минимум поддерживается локаль `ru` (в дальнейшем возможно расширение).

---

### 4. Требования по безопасности

- **Расположение кода:**
  - весь код модуля (PHP, JS, шаблоны админки) размещается в `/local/modules/` и
    связанных ресурсных директориях (`local/templates`, если применимо).
- **Защита от CSRF:**
  - для всех AJAX‑запросов и форм обязателен вызов `check_bitrix_sessid()`.
- **Права доступа:**
  - перед выполнением бизнес‑действий модуль должен проверять права:
    - на уровне CRM через `CCrmAuthorizationHelper` либо эквивалентные механизмы;
    - на уровне модульных прав (роль администратора/настройщика модуля).
- **Валидация входных данных:**
  - проверка корректности и принадлежности `ENTITY_ID`, `ELEMENT_ID`, `FIELD_ID`, `USER_ID`;
  - защита от подмены контекста (например, запрет доступа к чужим элементам).
- **Логирование ошибок:**
  - ошибки backend‑обработчиков записываются в лог (по стандарту Bitrix) без утечки
    конфиденциальных данных в интерфейс пользователя.

---

### 5. План расширения (будущие версии)

Следующие возможности не обязательны для первой версии, но должны учитываться в архитектуре:

1. **Условия отображения кнопки:**
   - возможность задавать правила видимости кнопки:
     - в зависимости от стадии сделки;
     - в зависимости от значений других полей CRM;
     - в зависимости от роли/прав пользователя.
2. **Интеграция с Роботами / бизнес‑процессами:**
   - возможность инициировать «виртуальное нажатие» кнопки из робота/бизнес‑процесса,
     с передачей контекста, аналогичного пользовательскому клику.
3. **Логирование нажатий:**
   - отдельная таблица `my_bpbutton_logs`:
     - фиксация факта нажатия (дата, пользователь, сущность, результат);
     - хранение статуса выполнения и, при необходимости, диагностического сообщения.

Архитектура backend (D7) и модель данных должны быть спроектированы так, чтобы эти
расширения могли быть добавлены без критического рефакторинга.
