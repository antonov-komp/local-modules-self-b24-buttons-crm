## Состояния кнопки и обработка ошибок

Документ сводит в одно место:
- все возможные состояния кнопки `bp_button_field`;
- визуальные эффекты для каждого состояния;
- типовые ошибки и ожидаемые реакции UI.

Ссылается на:
- `crm_card_button.md` (UX описания);
- `../api/ajax_controller.md` (коды ошибок и формат JSON);
- `../architecture/frontend_ui.md` (JS‑контракт).

---

### 1. Состояния кнопки

#### 1.1. Таблица состояний

| Состояние  | Описание                                      | CSS‑классы                                      | Пользовательский опыт                        |
|-----------|-----------------------------------------------|-------------------------------------------------|----------------------------------------------|
| `idle`    | Нормальное, готово к нажатию                 | `ui-btn ui-btn-primary` + JS‑класс кнопки       | Видит активную кнопку, может нажать          |
| `loading` | Запрос отправлен, ожидаем ответ              | `ui-btn ui-btn-primary ui-btn-wait`            | Видит индикатор загрузки, повторный клик блокируется |
| `disabled`| Кнопка недоступна по бизнес‑логике/настройкам| `ui-btn ui-btn-disabled` или снятие обработчика| Понимает, что действие недоступно            |
| `error`*  | Временное визуальное состояние при ошибке    | `idle` + опциональная краткая подсветка ошибки | Получает уведомление, кнопка остаётся кликабельной или временно отключённой |

\* Состояние `error` трактуется как краткое визуальное состояние поверх `idle`/`disabled` (анимация/подсветка по желанию).

#### 1.2. Переходы состояний

- `idle` → `loading` — первый клик пользователя.
- `loading` → `idle` — успешный ответ от контроллера и запуск SidePanel.
- `loading` → `idle`/`error` — при ошибке (всегда снимаем `ui-btn-wait`).
- `idle` → `disabled` — при смене настроек (ACTIVE = N) или жёстких бизнес‑ограничениях.
- `disabled` → `idle` — при повторной активации кнопки через реестр.

### 2. Карта состояний и визуала

#### 2.1. Состояние `idle`

- **Классы:** `ui-btn ui-btn-primary js-bp-button-field` (или другая допустимая вариация).
- **UX:**
  - курсор `pointer`;
  - чёткий, понятный текст (локализованный);
  - нет индикаторов загрузки.

#### 2.2. Состояние `loading`

- **Классы:** базовые классы + `ui-btn-wait`.
- **UX:**
  - визуальный индикатор ожидания (стандартный спиннер Bitrix в кнопке);
  - повторные клики игнорируются до получения ответа;
  - переход в это состояние должен происходить мгновенно (не более ~100 мс).

#### 2.3. Состояние `disabled`

- **Классы:** `ui-btn ui-btn-disabled` (вариант — не навешивать обработчик клика).
- **UX:**
  - приглушённый цвет;
  - курсор `default`;
  - при попытке кликнуть (если обработчик всё же стоит) — никакой реакции или показ короткого объяснения (по решению проекта).

#### 2.4. Состояние `error`

- **Классы:** как `idle`; возможна опциональная краткая подсветка (красная рамка/фон) на 300–500 мс.
- **UX:**
  - кнопка не зависает в `loading`;
  - пользователь получает уведомление с объяснением;
  - при повторном клике (если логика не запрещает) можно повторить попытку.

### 3. Коды ошибок и UX‑реакции

#### 3.1. Таблица кодов ошибок

| Код ошибки         | Типовая причина                               | Сообщение пользователю (пример)                           | UI‑реакция                                         |
|--------------------|-----------------------------------------------|-----------------------------------------------------------|----------------------------------------------------|
| `INVALID_SESSION`  | Истёкшая/отсутствующая сессия                 | «Сессия истекла. Обновите страницу и повторите попытку.»  | Кнопка → `idle`, показываем уведомление            |
| `ACCESS_DENIED`    | Нет прав на сущность/элемент                  | «Недостаточно прав для выполнения действия.»              | Кнопка → `idle` или `disabled`, уведомление        |
| `SETTINGS_NOT_FOUND`| Нет записи в `my_bpbutton_settings`         | «Кнопка не настроена. Обратитесь к администратору.»       | Кнопка → `idle`, уведомление                       |
| `BUTTON_INACTIVE`  | В настройках кнопка выключена (`ACTIVE = N`) | «Действие недоступно. Кнопка отключена администратором.»  | Кнопка → `disabled`, уведомление (однократно)      |
| `INTERNAL_ERROR`   | Внутренняя ошибка/исключение                  | «Произошла ошибка. Попробуйте позже или обратитесь к администратору.» | Кнопка → `idle`, уведомление            |

Тексты конкретных сообщений должны храниться в языковых файлах и соответствовать этому смыслу.

#### 3.2. Поведение фронтенда при ошибке

Общий алгоритм:

1. Снять `ui-btn-wait` с кнопки.
2. Прочитать `error.code` и `error.message` из JSON‑ответа.
3. Вызвать:

```javascript
BX.UI.Notification.Center.notify({
  content: error.message,
  autoHideDelay: 5000
});
```

4. При `BUTTON_INACTIVE`:
   - опционально добавить `ui-btn-disabled` и больше не переводить кнопку в `loading`.

### 4. Связь с use cases

- UC‑002 (ежедневная работа менеджера):
  - нормальный цикл `idle → loading → idle` без ошибок.
- UC‑003 (кнопка не настроена):
  - `idle → loading → idle + уведомление` с кодом `SETTINGS_NOT_FOUND`.
- UC‑004 (отключение кнопок админом):
  - после изменения `ACTIVE = N` при следующем клике:
    - `idle → loading → disabled` + уведомление с `BUTTON_INACTIVE`.

### 5. Чек‑лист по состояниям и ошибкам

1. **Состояния:**
   - [ ] Все четыре состояния (`idle`, `loading`, `disabled`, `error`) корректно обрабатываются фронтендом.
   - [ ] Нет сценариев, когда кнопка остаётся в `loading` навсегда.

2. **Ошибки:**
   - [ ] Для каждого кода ошибки (`INVALID_SESSION`, `ACCESS_DENIED`, `SETTINGS_NOT_FOUND`, `BUTTON_INACTIVE`, `INTERNAL_ERROR`) определено чёткое сообщение и UI‑реакция.
   - [ ] Сообщения локализованы и не содержат технических деталей.

3. **UX:**
   - [ ] Пользователь всегда понимает, что произошло (ошибка/отказ/отключение).
   - [ ] После ошибки пользователь либо может повторить действие, либо явно видит, что кнопка недоступна.
