# TASK-002: Административный реестр полей «Кнопки БП»

**Дата создания:** 2026-03-16 (UTC+3, Брест)  
**Статус:** Новая  
**Приоритет:** Средний  
**Исполнитель:** Bitrix24 Программист (коробка, D7 + Vanilla JS)

## Описание

Задача описывает реализацию административного реестра для пользовательских полей типа `bp_button_field`:

- Автоматическая регистрация поля в реестре при его создании в CRM.
- Настройка логики: URL обработчика, заголовок окна, ширина SidePanel и др.

Реестр должен позволять администратору видеть все поля `bp_button_field`, быстро находить нужные кнопки по сущности/названию и настраивать их поведение без правок кода.

## Контекст

- Бизнес‑контекст и цели модуля описаны в `../local_button.md`.
- Архитектура backend‑части (D7, сервисы, контроллеры) описана в:
  - `../architecture/backend_d7.md`;
  - `../architecture/data_model.md`;
  - `../architecture/security.md`.
- UX‑ожидания по админ‑странице реестра зафиксированы в `../ui_ux/admin_page.md`.
- Текущее задание продолжает `TASK-001-user-type` и опирается на созданную таблицу `my_bpbutton_settings`.

## Модули и компоненты

- `local/modules/my.bpbutton/lib/Internals/SettingsTable.php`  
  — D7 ORM‑сущность `my_bpbutton_settings`.
- `local/modules/my.bpbutton/lib/Service/ButtonService.php`  
  — сервисный слой (методы для работы с настройками кнопок, в т.ч. для auto‑регистрации).
- `local/modules/my.bpbutton/admin/bpbutton_list.php`  
  — административный реестр («список») настроек полей `bp_button_field`.
- `local/modules/my.bpbutton/admin/bpbutton_list_ajax.php` (опционально)  
  — AJAX‑обработчики для списка (быстрые правки, массовые операции).
- `local/modules/my.bpbutton/admin/menu.php`  
  — пункт меню «Настройки → Мои модули → Кнопки БП».
- `local/modules/my.bpbutton/lang/ru/admin/bpbutton_list.php`  
  — языковые строки для таблицы, фильтров и форм.
- `local/modules/my.bpbutton/lang/ru/options.php` (если используются глобальные опции).

## Ступенчатые подзадачи

1. **Подготовка структуры и меню**
   - Добавить пункт меню «Кнопки БП» в `admin/menu.php`:
     - раздел «Настройки → Мои модули»;
     - проверка прав модуля (`$APPLICATION->GetGroupRight('my.bpbutton')`).
   - Создать каркас файла `admin/bpbutton_list.php`:
     - подключение пролога/эпилога;
     - проверка прав администратора модуля.

2. **Реализация списка (реестра) на основе `SettingsTable`**
   - Использовать стандартный `CAdminList` / `CAdminUiList`:
     - источник данных — D7 ORM (`SettingsTable::getList()`).
   - Определить набор колонок в соответствии с `../ui_ux/admin_page.md`:
     - ID, поле (название/код), сущность (`ENTITY_ID`), `HANDLER_URL`, `TITLE`, `WIDTH`, `ACTIVE`, `UPDATED_AT`.
   - Реализовать сортировку и пагинацию.

3. **Реализация фильтрации**
   - Добавить фильтр по:
     - `ENTITY_ID` (тип сущности);
     - `ACTIVE` (`Y`/`N`);
     - части названия поля или его кода;
     - части `HANDLER_URL`.
   - Настроить сохранение состояния фильтра между запросами.

4. **Форма редактирования настроек**
   - Реализовать форму редактирования записи `my_bpbutton_settings`:
     - `FIELD_ID` / название поля — только для чтения;
     - `ENTITY_ID` — только для чтения или выбор из списка (по требованиям);
     - `HANDLER_URL`, `TITLE`, `WIDTH`, `ACTIVE` — редактируемые поля.
   - Валидация:
     - проверка формата URL (минимальная, без строгих ограничений);
     - проверка длины строк;
     - проверка допустимых значений ширины (число или строка с `%`).

5. **Быстрые действия и массовые операции**
   - Добавить возможность:
     - активировать/деактивировать несколько записей разом;
     - (опционально) удалить настройки (с подтверждением).
   - Реализовать обработку действий через:
     - POST‑форму списка или отдельный `bpbutton_list_ajax.php`.

6. **Интеграция с авто‑регистрацией (из `TASK-001`)**
   - Убедиться, что:
     - при создании нового поля `bp_button_field` запись **автоматически** появляется в реестре;
     - при удалении поля запись исчезает (через обработку событий `OnAfterUserFieldAdd` / `OnUserFieldDelete`).

7. **Локализация и UX‑тонкости**
   - Вынести все надписи в языковые файлы:
     - названия колонок, подписей полей, сообщений о сохранении/ошибках.
   - Согласовать внешний вид с `../ui_ux/admin_page.md`:
     - понятные заголовки колонок;
     - дружелюбные тексты уведомлений.

## API-методы

- В рамках данной задачи используются только:
  - D7‑методы ORM `SettingsTable` (`getList`, `update`, `delete`, и т.п.);
  - admin‑API Bitrix (`CAdminList`, `CAdminFilter`, `CAdminUiList`, `CAdminContextMenu`).
- Внешних REST‑методов Bitrix24 **не требуется**.

## Тестирование

- **Функциональные проверки:**
  - создать несколько полей `bp_button_field` для разных сущностей CRM и убедиться, что они появляются в реестре;
  - отфильтровать реестр по сущности и активности, проверить корректность выборки;
  - отредактировать настройки (URL, заголовок, ширину), сохранить и убедиться, что изменения отражаются в БД.
- **Проверка прав:**
  - открыть раздел под администратором — раздел доступен;
  - открыть раздел под пользователем без прав администратора модулей — доступ должен быть запрещён.
- **UX‑проверки:**
  - проверить отображение в светлой/тёмной теме;
  - проверить удобство фильтров и понятность заголовков колонок;
  - убедиться, что при ошибках валидации пользователь видит понятные сообщения.
