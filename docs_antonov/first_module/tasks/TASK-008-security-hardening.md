# TASK-008: Security‑review и hardening модуля «BP Button Field»

**Дата создания:** 2026-03-16 (UTC+3, Брест)  
**Статус:** Новая  
**Приоритет:** Высокий  
**Исполнитель:** Bitrix24 Программист (коробка, D7) / интегратор

---

## Описание

Задача описывает комплексную проверку безопасности и hardening для модуля «BP Button Field» на уровне:

- проверки корректного использования `check_bitrix_sessid()` и прав доступа во всех точках входа (AJAX‑контроллер, admin‑часть, обработчики событий);
- маскирования чувствительных данных в логах;
- ревью SQL/D7‑ORM на предмет производительности и защиты от инъекций.

Цель — привести модуль в состояние, соответствующее требованиям `architecture/security.md` и типовым best practices для коробочного Битрикс24.

## Ступенчатые подзадачи

### 1. Аудит `check_bitrix_sessid()` и CSRF‑защиты

1.1. **AJAX‑контроллер (`ButtonController`)**

- Проверить, что:
  - каждый action (`getConfigAction`, `logClickAction`, `getExtendedConfigAction` при наличии):
    - до выполнения основной логики проверяет сессию:

```php
if (!check_bitrix_sessid()) {
    // возврат ошибки INVALID_SESSION через единый формат
}
```

- При отсутствии/битой сессии:
  - возвращается код `INVALID_SESSION` в формате, описанном в `api/ajax_controller.md`;
  - не выполняется никакой бизнес‑логики и запросы к БД.

1.2. **Admin‑AJAX‑обработчики**

- Для inline‑операций в админке (например, смена `ACTIVE` в списке):
  - убедиться, что все POST‑запросы:
    - содержат `sessid`;
    - проверяются через `check_bitrix_sessid()`;
  - при провале проверки:
    - не выполняется изменение данных;
    - возвращается безопасное сообщение для пользователя.

1.3. **Формы в admin‑части**

- Для форм редактирования настроек (`bpbutton_edit.php` и др.):
  - проверить наличие стандартного скрытого поля `sessid` и проверок Bitrix;
  - при необходимости использовать `bitrix_sessid_post()` и встроенную проверку в admin‑форме.

---

### 2. Проверка прав доступа во всех точках

2.1. **Контроллер (`ButtonController`)**

- Для `getConfigAction`:
  - гарантировать, что после проверки сессии выполняется проверка прав на сущность:
    - через `CCrmAuthorizationHelper`/`CCrmPerms` или современные аналоги;
    - проверяется право на чтение соответствующей CRM‑сущности (`DEAL`, `LEAD` и т.п.) по `entityId` и `elementId`.
  - при отсутствии прав:
    - вернуть ошибку `ACCESS_DENIED` (формат — см. `api/ajax_controller.md`);
    - не раскрывать, существует ли элемент или какие‑либо его детали.

- Для `logClickAction` (если реализован):
  - аналогично проверять права на элемент;
  - не позволять залогировать событие по чужому элементу без прав.

2.2. **Admin‑раздел**

- `bpbutton_list.php` и `bpbutton_edit.php`:
  - проверять права модуля:

```php
$RIGHT = $APPLICATION->GetGroupRight('my.bpbutton');
if ($RIGHT < 'W') {
    $APPLICATION->AuthForm(GetMessage('ACCESS_DENIED'));
}
```

- Inline‑AJAX в админке:
  - кроме `check_bitrix_sessid()`, проверять:
    - что пользователь имеет права администратора модуля;
    - при отсутствии прав — возвращать JSON‑ошибку и не изменять данные.

2.3. **Обработчики событий (`EventHandler`)**

- Убедиться, что:
  - обработчики `OnAfterUserFieldAdd`/`OnUserFieldDelete`:
    - выполняют минимальный набор действий (создание/удаление записей в `SettingsTable`);
    - не выполняют операции, зависящие от прав конкретного пользователя (это системные события).
- При ошибках внутри обработчиков:
  - исключения перехватываются и логируются;
  - не ломают критичные процессы ядра (создание/удаление полей).

---

### 3. Маскирование чувствительных данных в логах

3.1. **Бизнес‑логи (`my_bpbutton_logs`)**

- Проверить, какие данные записываются в:
  - `STATUS`;
  - `MESSAGE`;
  - дополнительные поля (если есть).

- Убедиться, что:
  - не логируются напрямую:
    - токены, пароли, ключи вебхуков;
    - полные URL с параметрами авторизации;
    - чувствительные данные CRM (полные комментарии, персональные данные, кроме идентификаторов).

3.2. **Технические логи (файлы/системные)**

- Во всех местах, где используются:
  - `AddMessage2Log`;
  - `\Bitrix\Main\Diag\Debug`, `Logger` и т.п.:
    - убедиться, что в лог не попадают:
      - полные строки SQL с параметрами (допустимо, но рекомендуется аккуратно);
      - стеки с чувствительными данными;
    - при необходимости скрыть/обрезать:
      - части URL;
      - тела запросов/ответов внешних сервисов.

3.3. **Единая политика**

- Зафиксировать (в `logging/overview.md` и/или комментариях к коду) общую политику:
  - что можно, а что нельзя писать в `my_bpbutton_logs`;
  - какие поля использовать только для технических логов;
  - как маскировать конфиденциальные данные (например, частичная маска email/телефона/URL).

---

### 4. Ревью SQL/ORM на предмет безопасности и производительности

4.1. **ORM‑запросы (`SettingsTable`, `LogsTable`)**

- Проверить все вызовы:
  - `SettingsTable::getList()`, `::add()`, `::update()`, `::delete()` и т.п.;
  - `LogsTable::add()` и выборки для аналитики (если реализованы).

- Убедиться, что:
  - используются параметризованные условия (стандартный D7‑подход);
  - нет ручной сборки SQL‑строк с включением непроверенного пользовательского ввода;
  - индексы (`UX_BPBTN_FIELD`, `IX_BPBTN_ENTITY`, индексы логов) соответствуют типичным выборкам.

4.2. **Чистый SQL (если есть)**

- Найти все участки кода, где используется:
  - `$connection->query("SELECT ...")` с конкатенацией строк;
  - прямые `mysqli_*`/`mysql_*` (нежелательно в D7‑коде).

- При необходимости:
  - заменить на ORM или `query` с биндингом параметров;
  - исключить возможность SQL‑инъекций при подстановке пользовательских данных.

4.3. **Производительность**

- Для ключевых операций:
  - получение настроек по `FIELD_ID`;
  - выборка настроек для реестра (с фильтрами);
  - вставка логов;
  - аналитические выборки по логам (если реализованы UI/отчёты);

оценить:

- используются ли подходящие индексы;
- нет ли избыточных `JOIN`/полнопросмотров таблиц при типичном сценарии;
- нет ли лишних запросов в циклах (N+1).

---

### 5. Формат ошибок и утечка информации

5.1. **AJAX‑контроллер**

- Проверить, что:
  - при любых ошибках (исключения, проблемы в БД, нет настроек, нет прав) в JSON‑ответ:
    - не попадают stack trace, сырые сообщения исключений, фрагменты SQL;
  - возвращаемые коды и сообщения (`code`, `message`) соответствуют:
    - `api/ajax_controller.md`;
    - UX‑документу `ui_ux/states_and_errors.md`.

5.2. **Admin‑часть**

- При валидационных и технических ошибках в admin‑формах:
  - пользователю показываются понятные сообщения;
  - технические подробности (если нужно сохранить) уходят в серверные логи.

## Технические требования

- Везде, где возможен CSRF:
  - используется `check_bitrix_sessid()` или встроенная проверка admin‑формы;
  - при неуспехе запрос корректно завершается без изменений данных.
- Везде, где требуется проверка прав:
  - проверяются права модуля (admin‑часть) и/или права на CRM‑сущности (контроллер);
  - при отсутствии прав пользователь/клиент получает только безопасное сообщение.
- Логи:
  - не содержат конфиденциальных данных в открытом виде;
  - используются по назначению (бизнес‑ vs технические логи).
- SQL и D7‑ORM:
  - не содержат ручной конкатенации пользовательских данных в запросы;
  - опираются на индексы и проектную модель данных.

---

## Критерии приёмки

- [ ] Для всех action‑методов контроллера и admin‑AJAX‑обработчиков присутствует проверка `check_bitrix_sessid()` (где применимо).
- [ ] Для всех действий, затрагивающих CRM‑данные или настройки модуля, реализована проверка прав:
  - `getConfigAction` не отдаёт данные при отсутствии прав на элемент CRM;
  - admin‑операции недоступны пользователям без прав модуля.
- [ ] В логах (`my_bpbutton_logs` и системных) не сохраняются токены, пароли и другие чувствительные данные.
- [ ] Ни один JSON‑ответ контроллера не содержит технических деталей (SQL, stack trace).
- [ ] В коде модуля нет мест, где пользовательский ввод напрямую конкатенируется в SQL‑строки.
- [ ] Основные выборки по настройкам и логам используют индексы и не создают избыточную нагрузку при типичной эксплуатации.

## Тестирование

**Безопасность сессии:**

- Вызвать `getConfigAction` и admin‑AJAX‑операции:
  - с валидной сессией — запросы проходят;
  - с битой/отсутствующей сессией — возвращается `INVALID_SESSION` или эквивалент, данные не меняются.

**Права доступа:**

- Проверить сценарии под пользователем:
  - без прав на элемент CRM — контроллер возвращает `ACCESS_DENIED`, не раскрывая деталей;
  - без прав администратора модуля — admin‑раздел/inline‑операции недоступны.

**Логи:**

- Инициировать нажатия кнопок и ошибки:
  - убедиться, что записи в `my_bpbutton_logs` содержат только допустимый набор полей;
  - просмотреть системные логи и проверить отсутствие чувствительных данных.

**SQL/ORM:**

- Используя средства профилирования БД или логирования Bitrix:
  - убедиться, что нет подозрительных raw‑SQL с конкатенацией пользовательских данных;
  - проверить время выполнения типичных запросов по таблицам настроек и логов.
