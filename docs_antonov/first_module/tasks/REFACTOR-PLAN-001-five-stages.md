## Цель рефакторинга

- **Оптимизировать модуль глобально** — улучшить производительность, читаемость и поддерживаемость кода.
- **Разделить монолиты** — вынести логику из крупных файлов в отдельные классы и модули.
- **Улучшить развитие модуля** — создать чёткую структуру, упростить добавление новых фич и тестирование.

## Этапы рефакторинга

Каждый этап — **отдельная задача** с собственным TASK-документом. Ниже — поверхностное описание этапов.

---

### Этап 1: Разделение BpButtonUserType (UserField-слой)

**Задача:** `TASK-REF-001-userfield-split.md` (создать)

**Суть:** Класс `BpButtonUserType` выполняет слишком много обязанностей. Выделить:

- **ViewRenderer** (или `ButtonHtmlRenderer`) — генерация HTML кнопки для view/edit/admin-list.
- **SettingsResolver** (или расширить `ButtonService`) — получение настроек по `FIELD_ID` с кешированием.
- Оставить в `BpButtonUserType` только: описание типа, делегирование в рендерер и сервис.

**Результат:** `BpButtonUserType` — тонкая обёртка (~150–200 строк), остальная логика — в `lib/UserField/` или `lib/Service/`.

---

### Этап 2: Разделение админки (admin-слой)

**Задача:** `TASK-REF-002-admin-split.md` (создать)

**Суть:** Файл `bpbutton_list.php` объединяет список, форму редактирования, AJAX-обработку и валидацию. Разделить на:

- **bpbutton_list.php** — только список (грид, фильтры, меню). Подключение формы через include/компонент.
- **bpbutton_edit.php** (или форма в отдельном шаблоне) — форма редактирования настроек кнопки.
- **SettingsFormService** (или `AdminFormHandler`) — валидация, сохранение, подготовка данных для формы.
- AJAX-обработка — перенести в `ButtonController` или отдельный `AdminController`.

**Результат:** Админка — модульная, каждый файл < 300 строк, логика вынесена в сервисы.

---

### Этап 3: Рефакторинг JS-слоя (frontend)

**Задача:** `TASK-REF-003-js-split.md` (создать)

**Суть:** `button.js` содержит инициализацию, запросы к API, открытие SidePanel, управление состояниями и уведомлениями. Разделить на:

- **button.init.js** — инициализация, делегирование событий.
- **button.api.js** — запросы к `getConfigAction`, обработка ответов.
- **button.sidepanel.js** — логика открытия SidePanel, параметры.
- **button.state.js** — переключение состояний (loading, error, success).
- Сборка через `button.js` как entry point или через `CJSCore::RegisterExt` с несколькими файлами.

**Результат:** JS-модули с чёткой ответственностью, упрощённое тестирование и доработка.

---

### Этап 4: Усиление сервисного слоя и API

**Задача:** `TASK-REF-004-service-layer.md` (создать)

**Суть:** Унифицировать и усилить backend-слой:

- **ButtonService** — центральная точка для конфигурации кнопки (CRM), логирование кликов (если есть).
- **SettingsFormService** (новый) — CRUD для настроек в админке, валидация полей.
- **ButtonController** — только маршрутизация и вызов сервисов, без бизнес-логики.
- Единый формат ошибок и ответов API.
- Вынести дублирующую логику из `BpButtonUserType` и `bpbutton_list.php` в сервисы.

**Результат:** Чёткое разделение: Controller → Service → ORM, без дублирования.

---

### Этап 5: Структура модуля и документация

**Задача:** `TASK-REF-005-structure-docs.md` (создать)

**Суть:** Зафиксировать итоговую структуру и обновить документацию:

- **README модуля** — назначение, установка, основные точки входа.
- **Архитектурная схема** — слои (UserField, Admin, Controller, Service, ORM), потоки данных.
- **Обновить** `module_structure/filesystem.md`, `architecture/backend_d7.md` под новую структуру.
- Проверить соответствие namespaces, путей, зависимостей.
- Чек-лист для новых разработчиков.

**Результат:** Модуль документирован, структура прозрачна, упрощён онбординг.
