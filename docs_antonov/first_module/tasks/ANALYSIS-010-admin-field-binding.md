# ANALYSIS-010: Анализ связи раздела администрирования с полями bp_button_field

**Дата создания:** 2026-03-16 (UTC+3, Брест)  
**Роль:** Технический писатель  
**Статус:** Анализ завершён  
**Цель:** Выявить причины, по которым настройки из раздела «Кнопки БП» не применяются к полям в CRM

---

## Резюме проблем (по обратной связи пользователя)

1. **Заголовок не меняется** — поле отображается как «КНОПКА» независимо от настроек
2. **Размер не меняется** — параметр WIDTH не применяется
3. **URL обработчик не применяется** — HANDLER_URL из админки не используется
4. **Раздел администрирования не связан с полями** — связь только по ID и Имени

---

## 1. Критическая ошибка: неверная ссылка «Настроить кнопку»

### Место

`local/modules/my.bpbutton/lib/UserField/BpButtonUserType.php`, метод `getSettingsHTML` (стр. 315–317):

```php
$settingsUrl = '/bitrix/admin/my_bpbutton_bpbutton_list.php?lang=' . LANGUAGE_ID . '&ID=' . $fieldId . '&action=edit';
```

### Суть проблемы

- В ссылку передаётся **`$fieldId`** — это **ID пользовательского поля** (из `b_user_field`)
- Форма редактирования в `admin/bpbutton_list.php` ожидает параметр **`ID`** как **первичный ключ записи в `my_bpbutton_settings`** (автоинкремент)

Структура таблицы `my_bpbutton_settings`:

| Колонка   | Описание                          |
|-----------|-----------------------------------|
| `ID`      | PRIMARY KEY, автоинкремент (1, 2, 3…) |
| `FIELD_ID`| ID пользовательского поля (123, 124…) |

При создании поля с ID=123 в `b_user_field` создаётся запись в `my_bpbutton_settings` с `ID=1`, `FIELD_ID=123`.

Ссылка `?ID=123&action=edit` приводит к вызову:

```php
$settingsRow = SettingsTable::getByPrimary(123)->fetch();
```

Метод ищет запись с **ID=123**, тогда как нужная запись имеет **ID=1**. Результат: **«Record not found»** или открытие неверной записи.

### Последствия

- Пользователь не может перейти к настройкам кнопки из формы создания/редактирования поля
- Даже при прямом заходе в реестр: клик «Редактировать» по строке передаёт корректный `$row['ID']` (ID записи SettingsTable), поэтому из списка форма открывается правильно
- Но ссылка «Настроить кнопку» в форме настроек поля **всегда неверна**

### Рекомендуемое решение

Вариант A: искать по `FIELD_ID` при наличии параметра `FIELD_ID`:

```php
// В admin/bpbutton_list.php, блок формы редактирования:
$id = (int)$request->get('ID');
$fieldIdParam = (int)$request->get('FIELD_ID');

if ($fieldIdParam > 0) {
    // Переход из формы настроек поля — ищем по FIELD_ID
    $settingsRow = SettingsTable::getList([
        'filter' => ['=FIELD_ID' => $fieldIdParam],
        'limit' => 1
    ])->fetch();
    if ($settingsRow) {
        $id = (int)$settingsRow['ID'];
    }
}
```

И в `getSettingsHTML`:

```php
$settingsUrl = '...&FIELD_ID=' . $fieldId . '&action=edit';
```

Вариант B: в `getSettingsHTML` получать ID записи SettingsTable по FIELD_ID и подставлять его в ссылку.

## 2. Текст кнопки не берётся из настроек

### Место

`local/modules/my.bpbutton/lib/UserField/BpButtonUserType.php`, метод `getPublicViewHTML` (стр. 164–166):

```php
$buttonText = Loc::getMessage('BPBUTTON_USER_TYPE_BUTTON_TEXT')
    ?: Loc::getMessage('BPBUTTON_USER_TYPE_NAME')
    ?: 'Кнопка';
```

### Суть проблемы

- Текст кнопки **всегда** берётся из языковых файлов
- **Нет обращения к `SettingsTable`** при рендере
- В `SettingsTable` есть поле `TITLE`, но оно используется только для **заголовка SidePanel** при открытии окна, а не для текста кнопки

### Семантика полей в SettingsTable

| Поле        | Назначение                         | Где используется                    |
|-------------|------------------------------------|-------------------------------------|
| `TITLE`     | Заголовок окна SidePanel           | `ButtonService::getSidePanelConfig` |
| `HANDLER_URL` | URL обработчика                 | `ButtonService::getSidePanelConfig` |
| `WIDTH`     | Ширина SidePanel                   | `ButtonService::getSidePanelConfig` |

Текста кнопки в настройках **нет**. Есть два варианта:

1. Добавить поле `BUTTON_TEXT` (или `LABEL`) в `SettingsTable`
2. Использовать `TITLE` и для текста кнопки, и для заголовка SidePanel

### Рекомендуемое решение

- Добавить поле `BUTTON_TEXT` в `SettingsTable` (или переиспользовать `TITLE` по соглашению)
- В `getPublicViewHTML` при наличии `field['ID']` загружать настройки из `SettingsTable` и подставлять `BUTTON_TEXT` (или `TITLE`) вместо `Loc::getMessage`
- Учесть кеширование/производительность: при рендере множества полей не делать N+1 запросов

## 3. «Заголовок не меняется» — уточнение

Возможны два значения «заголовка»:

### 3.1. Заголовок поля в форме CRM (label)

- Хранится в `b_user_field`: `LIST_COLUMN_LABEL`, `EDIT_FORM_LABEL`
- Задаётся при создании/редактировании поля в настройках CRM
- Модуль `my.bpbutton` **не управляет** этими полями
- Если пользователь имеет в виду именно это — проблема в интерфейсе Bitrix или в том, что настройки поля не сохраняются

### 3.2. Текст на самой кнопке

- См. п. 2 — сейчас берётся только из `Loc::getMessage`
- Нужна привязка к настройкам из `SettingsTable`

## 4. Размер (WIDTH) и URL обработчика «не применяются»

### Цепочка применения

1. Пользователь нажимает кнопку в карточке CRM
2. JS (`button.js`) отправляет AJAX: `entityId`, `elementId`, `fieldId`, `sessid`
3. `ButtonController::getConfigAction` → `ButtonService::getSidePanelConfig`
4. `getSidePanelConfig` ищет запись по `FIELD_ID`, проверяет `ACTIVE`, `HANDLER_URL`
5. При успехе возвращает `url`, `title`, `width`
6. JS открывает SidePanel с этими параметрами

### Условия, при которых настройки не применяются

1. **`HANDLER_URL` пустой** — `getSidePanelConfig` возвращает ошибку «Кнопка не настроена», SidePanel не открывается → WIDTH и TITLE не используются
2. **Неверная ссылка «Настроить кнопку»** (п. 1) — пользователь не может открыть форму и сохранить URL/размер
3. **`fieldId` не передаётся или передаётся неверно** — кнопка в HTML должна иметь `data-field-id`. Если Entity Editor передаёт поле без `ID` или с другим ключом, `fieldId` будет 0, и настройки не найдутся

### Проверка передачи fieldId

В `BpButtonUserType::getPublicViewHTML`:

```php
$fieldId = (string)($field['ID'] ?? '');
// ...
'data-field-id="' . htmlspecialcharsbx($fieldId) . '"',
```

Структура `$field` зависит от контекста (Entity Editor, компонент, список). Нужно убедиться, что во всех режимах в `$field` есть `ID` пользовательского поля.

## 5. Раздел администрирования «не связан с полями»

### Текущая связь

- `SettingsTable.FIELD_ID` = `b_user_field.ID` — связь по ID поля
- При создании поля `bp_button_field` создаётся запись в `my_bpbutton_settings` (EventHandler)
- При удалении поля запись удаляется

### Проблемы отображения

#### 5.1. Колонка «Поле» в списке

В `admin/bpbutton_list.php` (стр. 321–334):

```php
if ($findFieldQuery !== '') {
    $runtime['UF'] = new Entity\ReferenceField(...);
    $filter[] = [...];
}
// ...
if (!empty($runtime['UF'])) {
    $select[] = 'UF_FIELD_NAME';
}
```

`UF` (join с `UserFieldTable`) добавляется **только при непустом** `find_field_query`. Без фильтра по полю `UF_FIELD_NAME` не выбирается, и в колонке «Поле» отображается только `ID=123` вместо названия поля.

**Решение:** всегда делать join с `UserFieldTable` для отображения названия поля (или подгружать его отдельно для каждой строки).

#### 5.2. Ссылка «Настроить кнопку»

См. п. 1 — ссылка ведёт на неверную запись.

## 6. Сводная таблица причин и решений

| Проблема                         | Причина                                      | Решение |
|----------------------------------|----------------------------------------------|---------|
| Ссылка «Настроить кнопку» неверна | Передаётся FIELD_ID как ID, ожидается PK     | Поддержать параметр FIELD_ID в форме редактирования или получать ID записи по FIELD_ID в getSettingsHTML |
| Текст кнопки «КНОПКА» не меняется | Текст только из Loc::getMessage              | Добавить поле BUTTON_TEXT (или использовать TITLE) и загружать его в getPublicViewHTML |
| Размер не применяется            | SidePanel не открывается при пустом HANDLER_URL; возможна ошибка в ссылке на настройки | Исправить ссылку (п. 1); убедиться, что HANDLER_URL сохраняется в нужную запись |
| URL обработчика не применяется   | Аналогично                                   | Исправить ссылку (п. 1) |
| В реестре только ID, без имени   | Join с UserFieldTable только при фильтре     | Всегда джойнить UserFieldTable для колонки «Поле» |

## 7. Рекомендуемый порядок исправлений

1. **Исправить ссылку «Настроить кнопку»** — без этого пользователь не может корректно настроить кнопку
2. **Всегда показывать название поля в реестре** — улучшить UX и понимание связи
3. **Добавить использование настроек для текста кнопки** — поле BUTTON_TEXT или TITLE в getPublicViewHTML
4. **Проверить передачу fieldId** — при необходимости добавить логирование и обработку разных форматов `$field`
