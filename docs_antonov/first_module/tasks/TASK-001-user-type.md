# TASK-001: Регистрация типа пользовательского поля `bp_button_field`

**Дата создания:** 2026-03-16 (UTC+3, Брест)  
**Статус:** В работе  
**Приоритет:** Высокий  
**Исполнитель:** Bitrix24 Программист (коробка, D7 + Vanilla JS)

## Описание

Задача описывает регистрацию и базовое поведение кастомного типа пользовательского поля `bp_button_field`, используемого для отображения кнопки в карточках CRM и смарт-процессах.

Тип поля должен отображаться в интерфейсе добавления пользовательских полей CRM, а при просмотре карточки вместо текстового значения выводиться в виде активной кнопки, запускающей JS‑расширение модуля.

## Контекст

- Бизнес‑требования сформулированы в `requirements.md` и master‑документе `local_button.md`.
- Архитектура backend‑части и модель данных описаны в `architecture/backend_d7.md` и `architecture/data_model.md`.
- Модуль реализуется в коробочной версии Битрикс24 в `/local/modules/my.bpbutton/`.

## Модули и компоненты

- `local/modules/my.bpbutton/lib/EventHandler.php`  
  — регистрация пользовательского типа `bp_button_field` через `OnUserTypeBuildList`,
  обработка `OnAfterUserFieldAdd` / `OnUserFieldDelete` (минимальный объём для данной задачи).
- `local/modules/my.bpbutton/lib/UserField/BpButtonUserType.php`  
  — класс пользовательского типа (view‑логика, HTML кнопки и интеграция с JS).
- `local/modules/my.bpbutton/lang/ru/lib/UserField/bpbuttonusertype.php`  
  — языковые сообщения для описания типа и надписей по умолчанию.
- `local/modules/my.bpbutton/install/index.php`  
  — регистрация обработчиков событий при установке модуля.

## Зависимости

- Используются API пользовательских полей Битрикс24 (user type).
- Требуется наличие таблицы `my_bpbutton_settings` и ORM‑класса `SettingsTable`
  (минимум — возможность сохранить запись по `FIELD_ID` при создании поля).
- Задачи:
  - `TASK-002-admin-grid` — админ‑интерфейс реестра настроек (последующая реализация);
  - `TASK-003-ajax-api` — AJAX‑контроллер и API для кнопки.

## Ступенчатые подзадачи

1. **Определить и создать класс пользовательского типа.** ✅  
   - Создан `BpButtonUserType` с описанием:
     - `USER_TYPE_ID = 'bp_button_field'`;
     - человеко‑читаемое описание через `Loc::getMessage` (`BPBUTTON_USER_TYPE_NAME`).
   - Реализованы базовые методы user type (описание, `GetDBColumnType`, `GetPublicViewHTML`,
     `GetAdminListViewHTML`, `GetEditFormHTML`).

2. **Реализовать обработчик `OnUserTypeBuildList`.** ✅  
   - В `EventHandler` реализован статический метод `onUserTypeBuildList`, возвращающий
     описание user type из `BpButtonUserType::getUserTypeDescription()`.
   - Обработчик события зарегистрирован в `install/index.php` через `RegisterModuleDependences`.

3. **Реализовать HTML‑представление кнопки для карточки CRM.** ✅  
   - В `BpButtonUserType` реализован метод публичного отображения `getPublicViewHTML`:
     - выводится кнопка с классами Bitrix UI `ui-btn ui-btn-primary` и маркером
       `js-bpbutton-field` для JS‑расширения;
     - добавлены `data-*` атрибуты: `data-entity-id`, `data-element-id`, `data-field-id`,
       `data-user-id`.
   - Подключение стилевого набора `ui.buttons` выполняется через `Extension::load`.

4. **Интегрировать с JS‑расширением (только контракт стороны PHP).** ✅  
   - В разметку кнопки добавлен CSS‑класс `js-bpbutton-field` и набор `data-*` атрибутов,
     позволяющих JS‑коду:
     - навесить обработчик клика;
     - переключать визуальные состояния кнопки (в т.ч. `state-wait`) в рамках последующих задач.
   - Реализация JS‑логики вынесена в отдельные задачи (`TASK-005-js-frontend` и др.).

5. **Подготовить связь с `SettingsTable` при создании поля.** ✅  
   - В обработчике `onAfterUserFieldAdd` проверяется, что создаётся поле с
     `USER_TYPE_ID = 'bp_button_field'`.
   - При создании такого поля через `SettingsTable::add()` создаётся запись в
     `my_bpbutton_settings` с `FIELD_ID` и дефолтными значениями настроек
     (`ACTIVE = 'Y'`, пустые `HANDLER_URL`, `TITLE`, `WIDTH`).

6. **Очистка настроек при удалении поля.** ✅  
   - В обработчике `onUserFieldDelete` при удалении поля типа `bp_button_field` по `FIELD_ID`
     находится и удаляется связанная запись в `my_bpbutton_settings`.

7. **Локализация.** ✅  
   - В языковом файле `lang/ru/lib/UserField/bpbuttonusertype.php` вынесены:
     - описание типа `BPBUTTON_USER_TYPE_NAME`;
     - текст кнопки по умолчанию `BPBUTTON_USER_TYPE_BUTTON_TEXT`.

## API-методы

- Используются стандартные механизмы пользовательских полей Bitrix (user type API).
- Внешних REST‑методов Bitrix24 в рамках задачи не требуется.

## Технические требования

- PHP 8.x, D7‑подход.
- Строгая регистрация модуля в `/local/modules/my.bpbutton/`.
- Код должен следовать стандартам Bitrix и PSR‑12.
- Все тексты — через `Loc::getMessage`.

## Критерии приёмки

- [ ] В интерфейсе добавления пользовательского поля CRM доступен новый тип `bp_button_field`.
- [ ] Поле `bp_button_field` можно создать для лида, сделки, контакта, компании и смарт‑процесса.
- [ ] При просмотре карточки вместо значения поля отображается кнопка Bitrix UI.
- [ ] Кнопка содержит корректные `data-*` атрибуты для JS (entity, element, field, user).
- [ ] При создании поля в `my_bpbutton_settings` появляется запись с соответствующим `FIELD_ID`.
- [ ] При удалении поля связанные настройки в `my_bpbutton_settings` удаляются.
- [ ] Все пользовательские тексты локализованы.

## Тестирование

- Создать поле `bp_button_field` для разных типов сущностей CRM и убедиться, что:
  - тип поля виден и выбирается в интерфейсе;
  - на карточке отображается кнопка с корректным текстом/стилем.
- Проверить, что в БД создаётся запись в `my_bpbutton_settings` с нужным `FIELD_ID`.
- Проверить удаление поля и очистку настроек.
