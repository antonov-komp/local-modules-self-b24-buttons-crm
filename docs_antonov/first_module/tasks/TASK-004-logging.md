# TASK-004: Логирование нажатий и аналитика использования кнопок

**Дата создания:** 2026-03-16 (UTC+3, Брест)  
**Статус:** Новая  
**Приоритет:** Средний  
**Исполнитель:** Bitrix24 Программист (коробка, D7 + Vanilla JS / PHP)

## Описание

Задача описывает реализацию подсистемы логирования для модуля «BP Button Field»:

- запись фактов нажатий на кнопки `bp_button_field` в таблицу `my_bpbutton_logs`;
- базовые средства аналитики (SQL‑запросы и, опционально, минимальный UI);
- интеграцию логирования с существующей архитектурой (контроллер, сервисный слой).

Логирование относится ко **второй версии** функционала и не является критичным для работы кнопок, но архитектура первой версии должна быть совместима с ним.

## Контекст

- Концепция логирования описана в `../local_button.md` и `../architecture/data_model.md`.
- Модель данных для логов (`my_bpbutton_logs`) описана в `../architecture/data_model.md` (DDL, индексы, примеры запросов).
- Безопасность и политика ошибок — в `../architecture/security.md`.
- API‑слой (включая зарезервированный `logClickAction`) — в `../api/ajax_controller.md`.

## Модули и компоненты

- `local/modules/my.bpbutton/lib/Internals/LogsTable.php`  
  — ORM‑сущность для таблицы `my_bpbutton_logs`.

- `local/modules/my.bpbutton/lib/Service/ButtonService.php`  
  — расширение сервисного слоя методами логирования:
  - `logClick(...)` — запись факта нажатия;
  - вспомогательные методы для аналитических запросов (по необходимости).

- `local/modules/my.bpbutton/lib/Controller/ButtonController.php`  
  — использование или реализация action‑метода:
  - `logClickAction(...)` — при необходимости отдельного эндпоинта;
  - либо внутренняя запись логов в рамках `getConfigAction`, если не требуется отдельный вызов.

- (Опционально) `local/modules/my.bpbutton/admin/bpbutton_logs.php`  
  — минимальный UI для просмотра логов по одной кнопке (просмотр последних событий).

## Зависимости

- От других задач:
  - `TASK-001-user-type` — кнопка уже отрисовывается и содержит нужные `data-*` атрибуты (entity/element/field/user);
  - `TASK-002-admin-grid` — есть реестр настроек, откуда можно переходить к аналитике по конкретной кнопке (опционально);
  - `TASK-003-ajax-api` — реализован или спроектирован контроллер `ButtonController` с `getConfigAction`.

- От архитектуры:
  - `../architecture/data_model.md` — структура `my_bpbutton_logs`;
  - `../architecture/security.md` — ограничения по логированию и доступу к данным логов.

## Ступенчатые подзадачи

1. **Реализовать ORM‑сущность `LogsTable`.**
   - Создать класс `My\BpButton\Internals\LogsTable`:
     - метод `getTableName()` → `my_bpbutton_logs`;
     - карта полей (`ID`, `SETTINGS_ID`, `FIELD_ID`, `ENTITY_ID`, `ELEMENT_ID`, `USER_ID`, `STATUS`, `MESSAGE`, `CREATED_AT`);
     - индексы, соответствующие DDL из `data_model.md`.

2. **Дополнить install/uninstall созданием/обработкой таблицы логов.**
   - При `DoInstall`:
     - создать таблицу `my_bpbutton_logs`, если её нет;
   - При `DoUninstall`:
     - определить политику (оставляем/удаляем таблицу, опциональный выбор через мастер деинсталляции).

3. **Расширить `ButtonService` методами логирования.**
   - Добавить методы:
     - `public function logClick(array $context, string $status, ?string $message = null): void`;
   - `context` должен включать:
     - `settingsId` (или `fieldId`), `entityId`, `elementId`, `userId`.
   - Внутри:
     - валидация входных данных;
     - вызов `LogsTable::add([...])`;
     - обработка/логирование исключений (не ломаем пользовательский сценарий при ошибках логирования).

4. **Интегрировать логирование в контроллер.**
   - Вариант A (минимальный): логировать только успешные нажатия внутри `getConfigAction`:
     - после успешного получения конфигурации вызвать `ButtonService::logClick(..., 'SUCCESS', null)`.
     - при ошибках — по необходимости логировать через технические логи, не в `my_bpbutton_logs`.
   - Вариант B (расширенный): реализовать отдельный `logClickAction`:
     - фронтенд после открытия SidePanel может отправлять отдельный запрос с `status` и `message`;
     - при этом `logClickAction` использует тот же сервис `logClick`.

5. **Подготовить набор аналитических запросов.**
   - Сформировать и задокументировать (в `../logging/overview.md`):
     - запросы по активности полей (`GROUP BY FIELD_ID`);
     - активность пользователей (`GROUP BY USER_ID`);
     - распределение статусов (`STATUS`) по периодам.

6. **(Опционально) UI для просмотра логов.**
   - Реализовать минимальную admin‑страницу:
     - просмотр последних N записей `my_bpbutton_logs` для конкретного `FIELD_ID`/`SETTINGS_ID`;
     - фильтры по дате, статусу, пользователю.
   - Добавить ссылку на эту страницу из реестра настроек (например, действие «Логи» возле записи).

7. **Локализация.**
   - Вынести все подписи/заголовки в языковые файлы:
     - названия статусов (`SUCCESS`, `ERROR`, `TIMEOUT` и т.п.);
     - заголовки колонок и фильтров в UI логов;
     - сообщения об ошибках/отсутствии данных.

## Поля лога (минимально необходимые)

Обязательные поля записи в `my_bpbutton_logs`:

- `ID` — автонумерация.
- `SETTINGS_ID` — связь с записью в `my_bpbutton_settings`.
- `FIELD_ID` — дублируется для удобства выборок.
- `ENTITY_ID` — тип CRM‑сущности (`DEAL`, `LEAD` и т.п.).
- `ELEMENT_ID` — ID конкретного элемента CRM.
- `USER_ID` — пользователь, нажавший кнопку.
- `STATUS` — результат (`SUCCESS`, `ERROR`, `TIMEOUT`, `CANCELLED` и т.п.).
- `MESSAGE` — краткое диагностическое сообщение (опционально).
- `CREATED_AT` — дата/время события.

Полный список и DDL — в `../architecture/data_model.md`.

## Отчёты и аналитические запросы

Задача должна обеспечить возможность выполнения как минимум следующих запросов (см. также `data_model.md`):

1. **Топ кнопок по количеству нажатий за период.**
   - `GROUP BY FIELD_ID` с подсчётом количества записей.

2. **Активность пользователей по нажатиям.**
   - `GROUP BY USER_ID` с фильтрацией по `STATUS = 'SUCCESS'` и периоду.

3. **Распределение статусов.**
   - `GROUP BY STATUS` по заданному интервалу времени.

4. **Подробный журнал по конкретной кнопке.**
   - выборка по `FIELD_ID`/`SETTINGS_ID` с сортировкой по `CREATED_AT DESC`.

Реализация самих отчётов (графики, внешний BI и т.п.) находится вне рамок задачи, но SQL‑запросы должны работать эффективно на целевых объёмах данных.

## Технические требования

- Использовать D7‑ORM для работы с таблицей `my_bpbutton_logs`.
- Логирование **не должно ломать** основной пользовательский сценарий:
  - при ошибках записи логов:
    - не прерывать выполнение `getConfigAction`;
    - фиксировать проблему только в технических логах.
- Соблюдать политику безопасности:
  - не логировать чувствительные данные целиком (токены, пароли и т.п.);
  - маскировать потенциально опасные части URL/сообщений.

## Критерии приёмки

- [ ] Таблица `my_bpbutton_logs` создаётся при установке модуля (если ещё не существует).
- [ ] Реализован `LogsTable` с полями, соответствующими описанию в `data_model.md`.
- [ ] `ButtonService` содержит метод для записи логов (`logClick`), использующий `LogsTable`.
- [ ] В контроллере реализовано (или интегрировано) логирование успешных нажатий (минимум для `SUCCESS`).
- [ ] Логирование ошибок не нарушает работу кнопки (ошибки записи логов не «падают» на пользователя).
- [ ] Есть документированные SQL‑запросы для базовой аналитики (количество нажатий по полям/пользователям/статусам).
- [ ] При включении UI‑страницы логов (если реализована) она корректно отображает данные и фильтрует их.

## Тестирование

- **Функциональные проверки:**
  - нажать кнопку `bp_button_field` в разных сущностях:
    - убедиться, что в `my_bpbutton_logs` появляется запись с корректным контекстом;
  - проверить, что при многократных нажатиях логируется каждый факт клика.

- **Проверка статусов:**
  - смоделировать успешный сценарий (`SUCCESS`);
  - смоделировать сценарий с ошибкой (`ERROR`), если используется расширенный `logClickAction` или логирование ошибок.

- **Нагрузочные проверки (по возможности):**
  - массовые нажатия (например, тестовым скриптом) и оценка производительности запросов к таблице логов.

- **Проверка безопасности:**
  - убедиться, что в `MESSAGE` и других полях не попадают сырые чувствительные данные.

   - Зафиксировать, **когда именно** писать лог:

4. **Интеграция с контроллером (`ButtonController`)**

- Логи должны быть пригодны для:
  - отладки инцидентов;
  - построения простых отчётов (по активности кнопок, пользователям, сущностям).

## Критерии приёмки
