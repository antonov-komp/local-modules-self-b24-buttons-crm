# TASK-004: Логирование нажатий кнопок и аудит

**Дата создания:** 2026-03-16 (UTC+3, Брест)  
**Статус:** Новая  
**Приоритет:** Средний  
**Исполнитель:** Bitrix24 Программист (коробка, D7)

## Описание

Задача описывает реализацию системы логирования для модуля:

- Проектирование и создание таблицы `my_bpbutton_logs`.
- Запись фактов нажатия кнопок, результата выполнения и контекста (пользователь, сущность, поле).
- Возможность дальнейшего анализа и отладки.

Требования берутся из раздела «План расширения» master-документа `local_button.md`.

## Контекст

- Общая концепция логирования и плана расширения описана в `../local_button.md`.
- Модель данных для таблицы логов (`my_bpbutton_logs`) зафиксирована в `../architecture/data_model.md`.
- Backend‑архитектура и сервисный слой описаны в `../architecture/backend_d7.md`.
- Взаимодействие с AJAX‑контроллером — в `TASK-003-ajax-api.md` и `../api/ajax_controller.md`.

## Модули и компоненты

- `local/modules/my.bpbutton/lib/Internals/LogsTable.php`  
  — D7 ORM‑сущность для таблицы `my_bpbutton_logs`.
- `local/modules/my.bpbutton/lib/Service/ButtonService.php`  
  — расширение сервисного слоя для записи логов (методы вида `logClick`, `logResult`).
- `local/modules/my.bpbutton/lib/Controller/ButtonController.php`  
  — при необходимости, использование action‑метода `logClickAction` для явного логирования (см. `TASK-003-ajax-api.md`).
- (в будущем) admin‑страницы для просмотра логов:
  - `local/modules/my.bpbutton/admin/bpbutton_logs.php` (или интеграция в существующий реестр).

## Зависимости

- От `TASK-002-admin-grid`:
  - наличие и структура реестра настроек (`my_bpbutton_settings`), к которому можно привязывать отчёты.
- От `TASK-003-ajax-api`:
  - наличие контроллера, из которого удобно вызывать логирование.
- От архитектуры:
  - `my_bpbutton_logs` должна соответствовать схеме в `architecture/data_model.md`.

## Ступенчатые подзадачи

1. **Реализация таблицы логов `my_bpbutton_logs`**
   - Создать D7‑класс `LogsTable` в `lib/Internals/LogsTable.php` согласно `data_model.md`:
     - поля: `ID`, `SETTINGS_ID`, `FIELD_ID`, `ENTITY_ID`, `ELEMENT_ID`, `USER_ID`, `STATUS`, `MESSAGE`, `CREATED_AT`;
     - индексы для быстрых выборок по полю, сущности, пользователю.
   - Обновить `install/index.php`:
     - добавить создание таблицы `my_bpbutton_logs` при установке модуля;
     - (опционально) предусмотреть удаление/сохранение логов при деинсталляции.

2. **Определить моменты логирования**
   - Зафиксировать, **когда именно** писать лог:
     - момент успешного получения конфигурации (`getConfigAction`) — статус `SUCCESS`;
     - ошибки при вызове контроллера (отказ в доступе, отсутствие настроек, внутренняя ошибка) — статусы `ACCESS_DENIED`, `SETTINGS_NOT_FOUND`, `INTERNAL_ERROR` и т.п.;
     - (опционально) событие фактического открытия SidePanel/iframe, если возможно отследить.
   - Минимальный набор полей для записи:
     - `FIELD_ID`, `ENTITY_ID`, `ELEMENT_ID`, `USER_ID`;
     - `STATUS` (SUCCESS/ERROR/…);
     - `MESSAGE` (краткое диагностическое описание ошибки или контекста);
     - `CREATED_AT` (время события).

3. **Расширение сервисного слоя `ButtonService`**
   - Добавить методы:
     - `public function logClick(array $context, string $status, ?string $message = null): void`;
   - Контекст (`$context`) должен включать:
     - `settingsId` (если есть), `fieldId`, `entityId`, `elementId`, `userId`.
   - Внутри метода:
     - подготовить и сохранить запись в `LogsTable`;
     - при ошибках записи логов — не ломать основной сценарий (только логировать в системный лог).

4. **Интеграция с контроллером (`ButtonController`)**
   - В `getConfigAction`:
     - при успешной выдаче конфигурации — опциональное логирование `SUCCESS` (по решению, чтобы не перегружать таблицу);
     - при ошибках (нет прав, нет настроек, внутренняя ошибка) — логирование со статусами (`ACCESS_DENIED`, `SETTINGS_NOT_FOUND`, `INTERNAL_ERROR`).
   - (Опционально) реализовать/задействовать `logClickAction`, который:
     - вызывается фронтендом после фактического открытия SidePanel;
     - пишет отдельный лог о том, что пользователь дошёл до интерфейса.

5. **Подготовка к будущему UI для логов**
   - Запроектировать, какие поля понадобятся для отображения логов в будущем admin‑интерфейсе:
     - по каким полям чаще всего фильтровать (датам, пользователям, сущностям, полям, статусам);
     - какие колонки будут ключевыми для разбора инцидентов.
   - Убедиться, что структура `my_bpbutton_logs` это поддерживает (индексы, типы полей).

## API-методы

- Явных публичных REST‑методов Bitrix24 не добавляется.
- Логирование выполняется через:
  - расширение `ButtonService` (внутренние вызовы);
  - (опционально) action `logClickAction` контроллера (`ButtonController`), описанный в `api/ajax_controller.md`.

## Технические требования

- Таблица `my_bpbutton_logs` создаётся при установке модуля и описана через D7‑ORM.
- Запись логов:
  - не должна ломать основной бизнес‑процесс (ошибки логирования только логируются);
  - не должна сохранять избыточные персональные данные — только необходимые идентификаторы и диагностические сообщения.
- Логи должны быть пригодны для:
  - отладки инцидентов;
  - построения простых отчётов (по активности кнопок, пользователям, сущностям).

## Критерии приёмки

- [ ] Таблица `my_bpbutton_logs` создана через миграцию/установку модуля и соответствует описанию в `architecture/data_model.md`.
- [ ] При ошибках в `getConfigAction` (нет прав, нет настроек, внутренняя ошибка) в `my_bpbutton_logs` появляется запись:
  - корректный `FIELD_ID`, `ENTITY_ID`, `ELEMENT_ID`, `USER_ID`;
  - осмысленный `STATUS` и `MESSAGE`.
- [ ] (Опционально) при успешных вызовах контроллера записываются логи со статусом `SUCCESS` (или это поведение отключаемо/ограничено).
- [ ] Ошибки при записи логов не влияют на выдачу конфигурации кнопки и не видны пользователю.

## Тестирование

- **Проверка структуры БД:**
  - убедиться, что таблица `my_bpbutton_logs` создана с нужными полями и индексами;
  - проверить базовые операции вставки/выборки через D7 (`LogsTable`).
- **Функциональные сценарии:**
  - вызвать `getConfigAction` для корректной кнопки и убедиться, что:
    - (при включённом логировании успехов) появляется запись со статусом `SUCCESS`;
  - смоделировать отсутствие настроек (`fieldId` без записи в `SettingsTable`) → запись с `STATUS = 'SETTINGS_NOT_FOUND'`;
  - смоделировать отсутствие прав → запись с `STATUS = 'ACCESS_DENIED'`.
- **Аналитические запросы:**
  - выполнить несколько SELECT‑запросов (см. `architecture/data_model.md`) и убедиться, что:
    - можно посчитать количество нажатий по полям за период;
    - можно получить топ‑пользователей по количеству нажатий.
